## نموذج القدرات العام

القدرات هي النموذج العام لـ **plugin الأصلية** داخل OpenClaw. إذ تسجّل كل
plugin أصلية في OpenClaw نفسها مقابل نوع قدرة واحد أو أكثر:

| القدرة                 | أسلوب التسجيل                                   | أمثلة على plugins                    |
| ---------------------- | ------------------------------------------------ | ------------------------------------ |
| الاستدلال النصي        | `api.registerProvider(...)`                      | `openai`, `anthropic`                |
| الكلام                 | `api.registerSpeechProvider(...)`                | `elevenlabs`, `microsoft`            |
| النسخ الفوري           | `api.registerRealtimeTranscriptionProvider(...)` | `openai`                             |
| الصوت الفوري           | `api.registerRealtimeVoiceProvider(...)`         | `openai`                             |
| فهم الوسائط            | `api.registerMediaUnderstandingProvider(...)`    | `openai`, `google`                   |
| توليد الصور            | `api.registerImageGenerationProvider(...)`       | `openai`, `google`, `fal`, `minimax` |
| توليد الموسيقى         | `api.registerMusicGenerationProvider(...)`       | `google`, `minimax`                  |
| توليد الفيديو          | `api.registerVideoGenerationProvider(...)`       | `qwen`                               |
| جلب الويب              | `api.registerWebFetchProvider(...)`              | `firecrawl`                          |
| البحث في الويب         | `api.registerWebSearchProvider(...)`             | `google`                             |
| القناة / المراسلة      | `api.registerChannel(...)`                       | `msteams`, `matrix`                  |

أما plugin التي لا تسجّل أي قدرات لكنها توفّر hooks أو أدوات أو
خدمات، فهي **plugin قديمة قائمة على hooks فقط**. وما يزال هذا النمط مدعومًا بالكامل.

### موقف التوافق الخارجي

لقد استقر نموذج القدرات في core ويُستخدم اليوم من قبل
plugins المجمعة/الأصلية، لكن توافق plugins الخارجية لا يزال يحتاج إلى معيار
أشد إحكامًا من "إنه مُصدّر، إذًا فهو مجمّد."

الإرشادات الحالية:

- **plugins الخارجية الحالية:** حافظ على تكاملات hooks وهي تعمل؛ وتعامل
  مع ذلك بصفته خط الأساس للتوافق
- **plugins المجمعة/الأصلية الجديدة:** فضّل التسجيل الصريح للقدرات بدلًا من
  الاختراقات الخاصة بالمورّد أو التصاميم الجديدة القائمة على hooks فقط
- **plugins الخارجية التي تعتمد تسجيل القدرات:** هذا مسموح، ولكن تعامل مع
  أسطح المساعدات الخاصة بالقدرات بوصفها قابلة للتطور ما لم تنص
  الوثائق صراحةً على أن العقد مستقر

قاعدة عملية:

- APIs تسجيل القدرات هي الاتجاه المقصود
- تبقى hooks القديمة المسار الأكثر أمانًا لتجنب الكسر بالنسبة إلى plugins الخارجية خلال
  الانتقال
- ليست كل المسارات الفرعية المصدّرة متساوية؛ ففضّل العقد الضيق الموثّق،
  لا صادرات المساعدات العرضية

### أشكال Plugin

يصنّف OpenClaw كل plugin محمّلة إلى شكل معيّن اعتمادًا على
سلوك التسجيل الفعلي لديها (وليس مجرد البيانات الوصفية الثابتة):

- **plain-capability** -- تسجّل نوع قدرة واحدًا بالضبط (على سبيل المثال
  plugin لموفّر فقط مثل `mistral`)
- **hybrid-capability** -- تسجّل أنواع قدرات متعددة (على سبيل المثال
  `openai` يمتلك الاستدلال النصي، والكلام، وفهم الوسائط، وتوليد
  الصور)
- **hook-only** -- تسجّل hooks فقط (مكتوبة النوع أو مخصصة)، بلا قدرات
  أو أدوات أو أوامر أو خدمات
- **non-capability** -- تسجّل أدوات أو أوامر أو خدمات أو مسارات، لكن بلا
  قدرات

استخدم `openclaw plugins inspect <id>` لرؤية شكل plugin وتفصيل
قدراتها. راجع [مرجع CLI](/cli/plugins#inspect) لمزيد من التفاصيل.

### Legacy hooks

لا يزال hook ‏`before_agent_start` مدعومًا بوصفه مسار توافق لـ
plugins القائمة على hooks فقط. ولا تزال plugins القديمة في العالم الحقيقي تعتمد عليه.

الاتجاه:

- أبقه يعمل
- وثّقه بوصفه قديمًا
- فضّل `before_model_resolve` لأعمال تجاوز النموذج/الموفّر
- فضّل `before_prompt_build` لأعمال تعديل prompt
- أزله فقط بعد تراجع الاستخدام الفعلي وإثبات تغطية fixtures لأمان الترحيل

### إشارات التوافق

عند تشغيل `openclaw doctor` أو `openclaw plugins inspect <id>`، قد ترى
إحدى هذه التسميات:

| الإشارة                    | المعنى                                                       |
| -------------------------- | ------------------------------------------------------------ |
| **config valid**           | يتم تحليل الإعدادات بشكل صحيح ويتم حل plugins               |
| **compatibility advisory** | تستخدم plugin نمطًا أقدم مدعومًا (مثل `hook-only`)          |
| **legacy warning**         | تستخدم plugin ‏`before_agent_start`، وهو مهجور              |
| **hard error**             | الإعدادات غير صالحة أو فشل تحميل plugin                     |

لن يؤدي `hook-only` ولا `before_agent_start` إلى كسر plugin لديك اليوم --
فـ `hook-only` مجرد تنبيه، و`before_agent_start` لا يؤدي إلا إلى
تحذير. وتظهر هذه الإشارات أيضًا في `openclaw status --all` و`openclaw plugins doctor`.

## نظرة عامة على البنية

يتكون نظام plugin في OpenClaw من أربع طبقات:

1. **البيان + الاكتشاف**
   يعثر OpenClaw على plugins المرشحة من المسارات المضبوطة، وجذور مساحة العمل،
   وجذور الامتدادات العامة، والامتدادات المجمعة. ويقرأ الاكتشاف
   بيانات `openclaw.plugin.json` الأصلية إضافةً إلى بيانات bundle المدعومة أولًا.
2. **التمكين + التحقق**
   يقرر core ما إذا كانت plugin المكتشفة مفعّلة، أو معطلة، أو محجوبة، أو
   محددة لفتحة حصرية مثل الذاكرة.
3. **التحميل أثناء التشغيل**
   تُحمّل plugins الأصلية لـ OpenClaw داخل العملية عبر jiti وتسجّل
   القدرات في سجل مركزي. أما الحزم المتوافقة فتُطبّع إلى سجلات داخل
   السجل من دون استيراد شيفرة وقت التشغيل.
4. **استهلاك السطح**
   يقرأ باقي OpenClaw السجل لكشف الأدوات، والقنوات، وإعداد
   الموفّرين، والhooks، ومسارات HTTP، وأوامر CLI، والخدمات.

أما بالنسبة إلى CLI الخاصة بـ plugin تحديدًا، فينقسم اكتشاف الأمر الجذري إلى مرحلتين:

- تأتي البيانات الوصفية وقت التحليل من `registerCli(..., { descriptors: [...] })`
- يمكن أن يبقى module الحقيقي الخاص بـ CLI في plugin كسولًا ويُسجّل عند أول استدعاء

ويُبقي ذلك شيفرة CLI المملوكة لـ plugin داخل plugin نفسها مع السماح لـ OpenClaw
بحجز أسماء الأوامر الجذرية قبل التحليل.

الحد التصميمي المهم:

- يجب أن يعمل الاكتشاف + التحقق من الإعدادات انطلاقًا من **بيانات البيان/المخطط**
  من دون تنفيذ شيفرة plugin
- يأتي السلوك الأصلي أثناء التشغيل من مسار `register(api)` في module الخاص بـ plugin

هذا الفصل يتيح لـ OpenClaw التحقق من الإعدادات، وشرح plugins المفقودة/المعطلة، و
بناء تلميحات UI/المخطط قبل تفعيل وقت التشغيل الكامل.

### Channel plugins وأداة الرسائل المشتركة

لا تحتاج Channel plugins إلى تسجيل أداة منفصلة للإرسال/التحرير/التفاعل من أجل
إجراءات الدردشة العادية. إذ يحتفظ OpenClaw بأداة `message` مشتركة واحدة داخل core،
وتملك channel plugins الاكتشاف والتنفيذ الخاصين بالقناة خلفها.

الحد الحالي هو:

- يملك core مضيف أداة `message` المشتركة، وربط prompt، وحفظ
  الجلسة/الخيط، وتوزيع التنفيذ
- تملك channel plugins اكتشاف الإجراءات المحددة، واكتشاف القدرات، وأي
  أجزاء مخطط خاصة بالقناة
- تملك channel plugins قواعد المحادثة الخاصة بجلسة الموفّر، مثل
  كيفية ترميز معرّفات المحادثة لمعرّفات الخيوط أو وراثتها من المحادثات
  الأصلية
- تنفّذ channel plugins الإجراء النهائي عبر action adapter الخاص بها

بالنسبة إلى channel plugins، يكون سطح SDK هو
`ChannelMessageActionAdapter.describeMessageTool(...)`. يتيح استدعاء الاكتشاف الموحد
هذا لـ plugin أن تعيد إجراءاتها المرئية، وقدراتها، ومساهماتها في
المخطط معًا حتى لا تنحرف هذه الأجزاء عن بعضها.

يمرر Core نطاق وقت التشغيل إلى خطوة الاكتشاف هذه. وتشمل الحقول المهمة:

- `accountId`
- `currentChannelId`
- `currentThreadTs`
- `currentMessageId`
- `sessionKey`
- `sessionId`
- `agentId`
- `requesterSenderId` الوارد الموثوق

وهذا مهم بالنسبة إلى plugins الحساسة للسياق. إذ يمكن للقناة إخفاء
إجراءات الرسائل أو كشفها استنادًا إلى الحساب النشط، أو الغرفة/الخيط/الرسالة الحالية، أو
هوية الطالب الموثوقة من دون ترميز فروع خاصة بالقناة داخل core في أداة `message`.

ولهذا السبب لا تزال تغييرات توجيه embedded-runner عملًا يخص plugin:
فالمشغّل مسؤول عن تمرير هوية الدردشة/الجلسة الحالية إلى حد
الاكتشاف في plugin حتى تكشف أداة `message` المشتركة السطح الصحيح
المملوك للقناة في الدور الحالي.

وبالنسبة إلى مساعدات التنفيذ المملوكة للقناة، ينبغي أن تُبقي plugins المجمعة وقت
التنفيذ داخل modules الامتداد الخاصة بها. فلم يعد core يملك أزمنة تشغيل
إجراءات رسائل Discord أو Slack أو Telegram أو WhatsApp تحت `src/agents/tools`.
كما أننا لا ننشر مسارات فرعية منفصلة من `plugin-sdk/*-action-runtime`، وينبغي أن
تستورد plugins المجمعة شيفرة وقت التشغيل المحلية الخاصة بها مباشرةً من
modules الامتداد المملوكة لها.

وينطبق الحد نفسه على seams الخاصة بـ SDK المسماة بحسب الموفّر عمومًا: لا ينبغي
لـ core أن يستورد أشرطة تسهيل خاصة بقنوات مثل Slack أو Discord أو Signal
أو WhatsApp أو الامتدادات المشابهة. وإذا احتاج core إلى سلوك ما، فإما أن يستهلك
شريط `api.ts` / `runtime-api.ts` الخاص بـ plugin المجمعة نفسها أو أن يرفع الحاجة
إلى قدرة عامة ضيقة في SDK المشتركة.

وبالنسبة إلى الاستطلاعات تحديدًا، هناك مسارا تنفيذ:

- `outbound.sendPoll` هو الخط الأساسي المشترك للقنوات التي تناسب
  نموذج الاستطلاع العام
- `actions.handleAction("poll")` هو المسار المفضل لدلالات الاستطلاع الخاصة بالقناة
  أو معلمات الاستطلاع الإضافية

يؤجل Core الآن تحليل الاستطلاع المشترك إلى ما بعد رفض توزيع استطلاع plugin
للإجراء، حتى تتمكن معالجات الاستطلاع المملوكة لـ plugin من قبول حقول
استطلاع خاصة بالقناة من دون أن يحظرها محلل الاستطلاع العام أولًا.

راجع [مسار التحميل](#load-pipeline) للتسلسل الكامل لبدء التشغيل.

## نموذج ملكية القدرات

يتعامل OpenClaw مع plugin الأصلية على أنها حد الملكية لـ **شركة**
أو **ميزة**، وليس بوصفها مجموعة عشوائية من التكاملات غير المرتبطة.

وهذا يعني:

- ينبغي أن تمتلك plugin الخاصة بالشركة عادةً جميع الأسطح الموجهة إلى OpenClaw لتلك الشركة
- ينبغي أن تمتلك plugin الخاصة بالميزة عادةً سطح الميزة الكامل الذي تقدمه
- ينبغي للقنوات أن تستهلك القدرات المشتركة من core بدلًا من إعادة تنفيذ
  سلوك الموفّر بشكل مخصص

أمثلة:

- تملك plugin المجمعة `openai` سلوك موفّر نموذج OpenAI وسلوك OpenAI
  للكلام + الصوت الفوري + فهم الوسائط + توليد الصور
- تملك plugin المجمعة `elevenlabs` سلوك الكلام الخاص بـ ElevenLabs
- تملك plugin المجمعة `microsoft` سلوك الكلام الخاص بـ Microsoft
- تملك plugin المجمعة `google` سلوك موفّر نموذج Google إضافةً إلى سلوك Google
  لفهم الوسائط + توليد الصور + البحث في الويب
- تملك plugin المجمعة `firecrawl` سلوك جلب الويب الخاص بـ Firecrawl
- تملك plugins المجمعة `minimax` و`mistral` و`moonshot` و`zai`
  واجهات فهم الوسائط الخلفية الخاصة بها
- plugin ‏`voice-call` هي plugin ميزة: فهي تملك نقل المكالمات، والأدوات،
  وCLI، والمسارات، وجسر Twilio لتدفق الوسائط، لكنها تستهلك القدرات المشتركة للكلام
  إضافةً إلى النسخ الفوري والصوت الفوري بدلًا من
  استيراد plugins الخاصة بالمورّدين مباشرةً

الحالة النهائية المقصودة هي:

- تعيش OpenAI في plugin واحدة حتى لو امتدت عبر نماذج النص، والكلام، والصور،
  والفيديو مستقبلًا
- يمكن لمورّد آخر أن يفعل الشيء نفسه لسطح منطقته الخاصة
- لا تهتم القنوات بأي plugin مورّد تملك الموفّر؛ فهي تستهلك عقد
  القدرة المشتركة الذي يكشفه core

وهذا هو التمييز الأساسي:

- **plugin** = حد الملكية
- **القدرة** = عقد core يمكن لعدة plugins تنفيذه أو استهلاكه

لذا إذا أضاف OpenClaw مجالًا جديدًا مثل الفيديو، فالسؤال الأول ليس
"أي موفّر يجب أن يرمّز معالجة الفيديو داخله؟" بل السؤال الأول هو: "ما
هو عقد قدرة الفيديو في core؟" وبمجرد وجود هذا العقد، تستطيع plugins الخاصة بالمورّدين
أن تسجّل نفسها مقابله، ويمكن لـ channel/feature plugins أن تستهلكه.

إذا لم تكن القدرة موجودة بعد، فإن الخطوة الصحيحة تكون عادةً:

1. تعريف القدرة المفقودة في core
2. كشفها عبر API/وقت تشغيل plugin بطريقة مكتوبة النوع
3. ربط القنوات/الميزات بهذه القدرة
4. السماح لـ vendor plugins بتسجيل التنفيذات

وهذا يُبقي الملكية صريحة ويمنع في الوقت نفسه السلوك في core الذي يعتمد على
مورّد واحد أو مسار شيفرة مخصص أحادي لـ plugin.

### طبقات القدرات

استخدم هذا النموذج الذهني عند تقرير موضع الشيفرة:

- **طبقة قدرات core**: التنسيق المشترك، والسياسات، وfallback، وقواعد
  دمج الإعدادات، ودلالات التسليم، والعقود المكتوبة النوع
- **طبقة vendor plugin**: APIs الخاصة بالمورّد، والمصادقة، وكتالوجات النماذج، و
  توليف الكلام، وتوليد الصور، وواجهات الفيديو الخلفية مستقبلًا، ونقاط استخدام
  النهاية
- **طبقة channel/feature plugin**: تكاملات Slack/Discord/voice-call/إلخ
  التي تستهلك قدرات core وتعرضها على سطح ما

على سبيل المثال، يتبع TTS هذا الشكل:

- يملك core سياسة TTS وقت الرد، وترتيب fallback، والتفضيلات، وتسليم القناة
- تملك `openai` و`elevenlabs` و`microsoft` تنفيذات التوليف
- تستهلك `voice-call` مساعد وقت تشغيل TTS الخاص بالاتصالات الهاتفية

ويجب تفضيل هذا النمط نفسه للقدرات المستقبلية.

### مثال على plugin شركة متعددة القدرات

ينبغي أن تبدو plugin الشركة متماسكة من الخارج. فإذا كان لدى OpenClaw عقود
مشتركة للنماذج، والكلام، والنسخ الفوري، والصوت الفوري، وفهم
الوسائط، وتوليد الصور، وتوليد الفيديو، وجلب الويب، والبحث في الويب،
فيمكن لمورّد أن يمتلك جميع أسطحه في مكان واحد:

```ts
import type { OpenClawPluginDefinition } from "openclaw/plugin-sdk/plugin-entry";
import {
  describeImageWithModel,
  transcribeOpenAiCompatibleAudio,
} from "openclaw/plugin-sdk/media-understanding";

const plugin: OpenClawPluginDefinition = {
  id: "exampleai",
  name: "ExampleAI",
  register(api) {
    api.registerProvider({
      id: "exampleai",
      // auth/model catalog/runtime hooks
    });

    api.registerSpeechProvider({
      id: "exampleai",
      // vendor speech config — implement the SpeechProviderPlugin interface directly
    });

    api.registerMediaUnderstandingProvider({
      id: "exampleai",
      capabilities: ["image", "audio", "video"],
      async describeImage(req) {
        return describeImageWithModel({
          provider: "exampleai",
          model: req.model,
          input: req.input,
        });
      },
      async transcribeAudio(req) {
        return transcribeOpenAiCompatibleAudio({
          provider: "exampleai",
          model: req.model,
          input: req.input,
        });
      },
    });

    api.registerWebSearchProvider(
      createPluginBackedWebSearchProvider({
        id: "exampleai-search",
        // credential + fetch logic
      }),
    );
  },
};

export default plugin;
```

ما يهم ليس أسماء المساعدات الدقيقة. بل المهم هو الشكل:

- plugin واحدة تملك سطح المورّد
- ولا يزال core يملك عقود القدرات
- وتستهلك القنوات وfeature plugins مساعدات `api.runtime.*`، لا شيفرة المورّد
- ويمكن لاختبارات العقد أن تؤكد أن plugin سجّلت القدرات التي
  تزعم أنها تملكها

### مثال على قدرة: فهم الفيديو

يتعامل OpenClaw بالفعل مع فهم الصور/الصوت/الفيديو بوصفه
قدرة مشتركة واحدة. وينطبق نموذج الملكية نفسه هنا:

1. يعرّف core عقد فهم الوسائط
2. تسجّل vendor plugins ‏`describeImage` و`transcribeAudio` و
   `describeVideo` حسب الاقتضاء
3. تستهلك channel وfeature plugins سلوك core المشترك بدلًا من
   الربط المباشر بشيفرة المورّد

وهذا يمنع ترسيخ افتراضات أحد الموفّرين حول الفيديو داخل core. إذ تملك plugin
سطح المورّد؛ بينما يملك core عقد القدرة وسلوك fallback.

كما أن توليد الفيديو يتبع بالفعل التسلسل نفسه: يملك core عقد
القدرة المكتوب النوع ومساعد وقت التشغيل، وتُسجّل vendor plugins
تنفيذات `api.registerVideoGenerationProvider(...)` مقابله.

هل تحتاج إلى قائمة طرح عملية ملموسة؟ راجع
[Capability Cookbook](/ar/plugins/architecture).

## العقود والإنفاذ

سطح API الخاص بـ plugin مكتوب النوع ومركزي عمدًا في
`OpenClawPluginApi`. ويحدد هذا العقد نقاط التسجيل المدعومة و
مساعدات وقت التشغيل التي يمكن أن تعتمد عليها plugin.

أهمية ذلك:

- يحصل مؤلفو plugins على معيار داخلي مستقر واحد
- يمكن لـ core رفض الملكية المكررة مثل قيام pluginين بتسجيل
  معرّف الموفّر نفسه
- يمكن لبدء التشغيل أن يعرض تشخيصات قابلة للتنفيذ للتسجيلات المشوهة
- يمكن لاختبارات العقد أن تفرض ملكية plugins المجمعة وتمنع الانحراف الصامت

هناك طبقتان من الإنفاذ:

1. **إنفاذ التسجيل أثناء التشغيل**
   يتحقق سجل plugin من التسجيلات أثناء تحميل plugins. أمثلة:
   معرّفات موفّرين مكررة، ومعرّفات موفّري كلام مكررة، وتسجيلات
   مشوهة تؤدي إلى تشخيصات plugin بدلًا من سلوك غير معرّف.
2. **اختبارات العقد**
   تُلتقط plugins المجمعة في سجلات عقود أثناء تشغيل الاختبارات بحيث
   يتمكن OpenClaw من تأكيد الملكية صراحةً. ويُستخدم هذا اليوم لموفّري
   النماذج، وموفّري الكلام، وموفّري البحث في الويب، وملكية
   التسجيل المجمعة.

الأثر العملي هو أن OpenClaw يعرف مسبقًا أي plugin تملك أي
سطح. وهذا يتيح لـ core والقنوات أن تتكوّن بسلاسة لأن الملكية معلنة، ومكتوبة النوع، وقابلة للاختبار بدلًا من أن تكون ضمنية.

### ما الذي يجب أن يكون ضمن العقد

عقود plugin الجيدة تكون:

- مكتوبة النوع
- صغيرة
- خاصة بالقدرة
- مملوكة لـ core
- قابلة لإعادة الاستخدام من قبل plugins متعددة
- قابلة للاستهلاك من القنوات/الميزات من دون معرفة بالمورّد

أما عقود plugin السيئة فهي:

- سياسات خاصة بالمورّد مخفية داخل core
- منافذ هروب خاصة بـ plugin واحدة تتجاوز السجل
- شيفرة قناة تصل مباشرةً إلى تنفيذ مورّد
- كائنات وقت تشغيل مخصصة ليست جزءًا من `OpenClawPluginApi` أو
  `api.runtime`

عند الشك، ارفع مستوى التجريد: عرّف القدرة أولًا، ثم دع plugins
تتصل بها.

## نموذج التنفيذ

تعمل plugins الأصلية لـ OpenClaw **داخل العملية نفسها** مع Gateway. وهي غير
معزولة. وتمتلك plugin الأصلية المحمّلة حد الثقة نفسه على مستوى العملية
الذي تمتلكه شيفرة core.

النتائج:

- يمكن لـ plugin الأصلية تسجيل أدوات، ومعالجات شبكة، وhooks، وخدمات
- يمكن لخلل plugin الأصلية أن يتسبب في تعطل gateway أو زعزعة استقرارها
- تعادل plugin أصلية خبيثة تنفيذ شيفرة عشوائية داخل عملية
  OpenClaw

تكون الحزم المتوافقة أكثر أمانًا افتراضيًا لأن OpenClaw يتعامل معها حاليًا
بوصفها حزم بيانات وصفية/محتوى. وفي الإصدارات الحالية، يعني هذا غالبًا
Skills المجمعة.

استخدم قوائم السماح ومسارات التثبيت/التحميل الصريحة من أجل plugins غير
المجمعة. وتعامل مع plugins مساحة العمل باعتبارها شيفرة وقت تطوير، لا
افتراضيات إنتاجية.

وبالنسبة إلى أسماء الحزم المجمعة في مساحة العمل، أبقِ معرّف plugin مرتكزًا في
اسم npm: ‏`@openclaw/<id>` افتراضيًا، أو لاحقة مكتوبة النوع معتمدة مثل
`-provider` أو `-plugin` أو `-speech` أو `-sandbox` أو `-media-understanding` عندما
تكشف الحزمة عمدًا دور plugin أضيق.

ملاحظة ثقة مهمة:

- `plugins.allow` يثق في **معرّفات plugin**، لا في مصدرها.
- تتعمد plugin مساحة العمل التي تحمل المعرّف نفسه لـ plugin مجمعة أن تظلّل
  النسخة المجمعة عندما تكون plugin مساحة العمل هذه مفعّلة/ضمن قائمة السماح.
- هذا سلوك طبيعي ومفيد للتطوير المحلي، واختبار الترقيعات، والإصلاحات العاجلة.

## حد التصدير

يصدر OpenClaw القدرات، لا وسائل الراحة التنفيذية.

اجعل تسجيل القدرات عامًا. وقلّص صادرات المساعدات غير التعاقدية:

- المسارات الفرعية الخاصة بمساعدات plugin المجمعة
- المسارات الفرعية الخاصة بسباكة وقت التشغيل غير المقصودة كـ API عامة
- مساعدات الراحة الخاصة بالمورّد
- مساعدات الإعداد/التهيئة التي تُعد تفاصيل تنفيذ

لا تزال بعض المسارات الفرعية لمساعدات plugins المجمعة موجودة في خريطة
تصدير SDK المولّدة لأغراض التوافق وصيانة plugins المجمعة. ومن الأمثلة الحالية:
`plugin-sdk/feishu` و`plugin-sdk/feishu-setup` و`plugin-sdk/zalo`,
`plugin-sdk/zalo-setup` وعدة seams من `plugin-sdk/matrix*`. تعامل مع هذه
بوصفها صادرات محجوزة لتفاصيل التنفيذ، لا بوصفها النمط الموصى به في SDK
لـ plugins الخارجية الجديدة.

## مسار التحميل

عند بدء التشغيل، يقوم OpenClaw تقريبًا بما يلي:

1. اكتشاف جذور plugins المرشحة
2. قراءة بيانات manifests الأصلية أو الحزم المتوافقة وبيانات package الوصفية
3. رفض المرشحين غير الآمنين
4. تطبيع إعدادات plugin ‏(`plugins.enabled`, `allow`, `deny`, `entries`,
   `slots`, `load.paths`)
5. تقرير التمكين لكل مرشح
6. تحميل modules الأصلية المفعلة عبر jiti
7. استدعاء hooks الأصلية `register(api)` (أو `activate(api)` — وهو اسم مستعار قديم) وجمع التسجيلات في سجل plugin
8. كشف السجل لأسطح الأوامر/وقت التشغيل

<Note>
`activate` اسم مستعار قديم لـ `register` — إذ يحل المحمّل أيّهما موجود (`def.register ?? def.activate`) ويستدعيه في النقطة نفسها. تستخدم جميع plugins المجمعة `register`؛ ففضّل `register` في plugins الجديدة.
</Note>

تحدث بوابات الأمان **قبل** تنفيذ وقت التشغيل. ويُحظر المرشحون
عندما يهرب entry من جذر plugin، أو يكون المسار قابلاً للكتابة من الجميع،
أو تبدو ملكية المسار مريبة بالنسبة إلى plugins غير المجمعة.

### سلوك manifest-first

البيان هو مصدر الحقيقة لطائرة التحكم. ويستخدمه OpenClaw من أجل:

- تحديد plugin
- اكتشاف القنوات/Skills/مخطط الإعدادات المعلنة أو قدرات bundle
- التحقق من `plugins.entries.<id>.config`
- تحسين تسميات/عناصر نائبة لـ Control UI
- عرض بيانات التثبيت/الكتالوج الوصفية

أما بالنسبة إلى plugins الأصلية، فإن module وقت التشغيل هو جزء طائرة البيانات. إذ يسجّل
السلوك الفعلي مثل hooks أو الأدوات أو الأوامر أو تدفقات الموفّر.

### ما الذي يخزّنه المحمّل مؤقتًا

يحتفظ OpenClaw بذاكرات مؤقتة قصيرة داخل العملية من أجل:

- نتائج الاكتشاف
- بيانات سجل manifest
- سجلات plugins المحمّلة

تقلل هذه الذاكرات المؤقتة من أعباء بدء التشغيل الفجائية وتكرار الأوامر.
ومن الآمن اعتبارها ذاكرات أداء قصيرة العمر، لا تخزينًا دائمًا.

ملاحظة أداء:

- اضبط `OPENCLAW_DISABLE_PLUGIN_DISCOVERY_CACHE=1` أو
  `OPENCLAW_DISABLE_PLUGIN_MANIFEST_CACHE=1` لتعطيل هذه الذاكرات المؤقتة.
- اضبط نوافذ cache باستخدام `OPENCLAW_PLUGIN_DISCOVERY_CACHE_MS` و
  `OPENCLAW_PLUGIN_MANIFEST_CACHE_MS`.

## نموذج السجل

لا تقوم plugins المحمّلة بتعديل متغيرات core العامة عشوائيًا مباشرةً. بل
تسجّل نفسها داخل سجل plugin مركزي.

يتتبع السجل ما يلي:

- سجلات plugin ‏(الهوية، والمصدر، والأصل، والحالة، والتشخيصات)
- الأدوات
- legacy hooks وtyped hooks
- القنوات
- الموفّرون
- معالجات Gateway RPC
- مسارات HTTP
- مسجلات CLI
- الخدمات الخلفية
- الأوامر المملوكة لـ plugin

ثم تقرأ ميزات core من هذا السجل بدلًا من التحدث إلى modules الخاصة بـ plugin
مباشرةً. وهذا يُبقي التحميل أحادي الاتجاه:

- module الخاصة بـ plugin -> التسجيل في السجل
- وقت تشغيل core -> استهلاك السجل

ويهم هذا الفصل من أجل سهولة الصيانة. فهو يعني أن معظم أسطح core لا
تحتاج إلا إلى نقطة تكامل واحدة: "اقرأ السجل"، لا "ضع حالة خاصة لكل
module plugin".

## callbacks الخاصة بربط المحادثة

يمكن للplugins التي تربط محادثة ما أن تتفاعل عند حسم الموافقة.

استخدم `api.onConversationBindingResolved(...)` لتلقي callback بعد الموافقة على طلب الربط أو رفضه:

```ts
export default {
  id: "my-plugin",
  register(api) {
    api.onConversationBindingResolved(async (event) => {
      if (event.status === "approved") {
        // A binding now exists for this plugin + conversation.
        console.log(event.binding?.conversationId);
        return;
      }

      // The request was denied; clear any local pending state.
      console.log(event.request.conversation.conversationId);
    });
  },
};
```

حقول حمولة callback:

- `status`: ‏`"approved"` أو `"denied"`
- `decision`: ‏`"allow-once"` أو `"allow-always"` أو `"deny"`
- `binding`: الربط المحسوم للطلبات الموافق عليها
- `request`: ملخص الطلب الأصلي، وتلميح الفصل، ومعرّف المرسل، و
  البيانات الوصفية للمحادثة

هذه callback مخصصة للإشعار فقط. وهي لا تغير من يُسمح له بربط
محادثة، كما أنها تعمل بعد انتهاء معالجة الموافقة داخل core.

## hooks وقت تشغيل الموفّر

أصبح لدى provider plugins الآن طبقتان:

- البيانات الوصفية في manifest: ‏`providerAuthEnvVars` للبحث الرخيص عن
  المصادقة عبر البيئة قبل تحميل وقت التشغيل، إضافةً إلى `providerAuthChoices` للحصول على
  تسميات التهيئة/اختيار المصادقة وبيانات أعلام CLI الوصفية قبل
  تحميل وقت التشغيل
- hooks وقت الإعدادات: ‏`catalog` / أو `discovery` القديمة إضافةً إلى `applyConfigDefaults`
- hooks وقت التشغيل: ‏`normalizeModelId`, `normalizeTransport`,
  `normalizeConfig`,
  `applyNativeStreamingUsageCompat`, `resolveConfigApiKey`,
  `resolveSyntheticAuth`, `shouldDeferSyntheticProfileAuth`,
  `resolveDynamicModel`, `prepareDynamicModel`, `normalizeResolvedModel`,
  `contributeResolvedModelCompat`, `capabilities`,
  `normalizeToolSchemas`, `inspectToolSchemas`,
  `resolveReasoningOutputMode`, `prepareExtraParams`, `createStreamFn`,
  `wrapStreamFn`, `resolveTransportTurnState`,
  `resolveWebSocketSessionPolicy`, `formatApiKey`, `refreshOAuth`,
  `buildAuthDoctorHint`, `matchesContextOverflowError`,
  `classifyFailoverReason`, `isCacheTtlEligible`,
  `buildMissingAuthMessage`, `suppressBuiltInModel`, `augmentModelCatalog`,
  `isBinaryThinking`, `supportsXHighThinking`,
  `resolveDefaultThinkingLevel`, `isModernModelRef`, `prepareRuntimeAuth`,
  `resolveUsageAuth`, `fetchUsageSnapshot`, `createEmbeddingProvider`,
  `buildReplayPolicy`,
  `sanitizeReplayHistory`, `validateReplayTurns`, `onModelSelected`

لا يزال OpenClaw يملك agent loop العام، وfailover، ومعالجة transcript،
وسياسة الأدوات. وهذه hooks هي سطح الامتداد للسلوك الخاص بالموفّر من دون
الحاجة إلى نقل استدلال مخصص بالكامل.

استخدم manifest ‏`providerAuthEnvVars` عندما يمتلك الموفّر بيانات اعتماد عبر البيئة
ينبغي أن تراها المسارات العامة للمصادقة/الحالة/منتقي النموذج من دون تحميل
وقت تشغيل plugin. واستخدم manifest ‏`providerAuthChoices` عندما ينبغي لأسطح
CLI الخاصة بالتهيئة/اختيار المصادقة أن تعرف معرّف اختيار الموفّر، وتسميات
المجموعات، وربط المصادقة البسيط ذي العلم الواحد، من دون تحميل وقت تشغيل الموفّر.
وأبقِ `envVars` الخاصة بوقت تشغيل الموفّر لتلميحات المشغّل مثل تسميات التهيئة أو
متغيرات إعداد OAuth client-id/client-secret.

### ترتيب hooks واستخدامها

بالنسبة إلى model/provider plugins، يستدعي OpenClaw hooks بهذا
الترتيب التقريبي. ويعد عمود "متى تستخدمه" دليل القرار السريع.

| #   | Hook                              | ما الذي يفعله                                                                          | متى تستخدمه                                                                                                                                 |
| --- | --------------------------------- | -------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | `catalog`                         | نشر إعداد الموفّر في `models.providers` أثناء توليد `models.json`                    | عندما يملك الموفّر كتالوجًا أو افتراضيات base URL                                                                                            |
| 2   | `applyConfigDefaults`             | تطبيق افتراضيات إعدادات عامة مملوكة للموفّر أثناء تجسيد الإعدادات                   | عندما تعتمد الافتراضيات على وضع المصادقة، أو البيئة، أو دلالات عائلة نماذج الموفّر                                                           |
| --  | _(built-in model lookup)_         | يحاول OpenClaw أولًا مسار السجل/الكتالوج العادي                                        | _(ليست hook خاصة بـ plugin)_                                                                                                                |
| 3   | `normalizeModelId`                | تطبيع الأسماء المستعارة القديمة أو التجريبية لـ model-id قبل البحث                    | عندما يملك الموفّر تنظيف الأسماء المستعارة قبل الحل القانوني للنموذج                                                                          |
| 4   | `normalizeTransport`              | تطبيع `api` / `baseUrl` الخاصة بعائلة الموفّر قبل تجميع النموذج العام                 | عندما يملك الموفّر تنظيف النقل لمعرّفات موفّرين مخصصة ضمن عائلة النقل نفسها                                                                  |
| 5   | `normalizeConfig`                 | تطبيع `models.providers.<id>` قبل حل وقت التشغيل/الموفّر                              | عندما يحتاج الموفّر إلى تنظيف إعدادات ينبغي أن تعيش مع plugin؛ كما تدعم مساعدات Google المجمعة هذه إدخالات Google المدعومة كخيار احتياطي   |
| 6   | `applyNativeStreamingUsageCompat` | تطبيق إعادة كتابة توافق استخدام البث الأصلي على موفّري الإعدادات                     | عندما يحتاج الموفّر إلى إصلاحات بيانات وصفية لاستخدام البث الأصلي مدفوعة بنقطة النهاية                                                        |
| 7   | `resolveConfigApiKey`             | حل مصادقة علامة البيئة لموفّري الإعدادات قبل تحميل مصادقة وقت التشغيل                 | عندما يمتلك الموفّر حلًا لمفتاح API عبر علامة البيئة؛ كما يملك `amazon-bedrock` محلل AWS مدمجًا هنا أيضًا                                     |
| 8   | `resolveSyntheticAuth`            | كشف مصادقة محلية/مستضافة ذاتيًا أو مدعومة بالإعدادات من دون حفظ نص صريح             | عندما يمكن للموفّر العمل بعلامة بيانات اعتماد اصطناعية/محلية                                                                                 |
| 9   | `shouldDeferSyntheticProfileAuth` | خفض أولوية العناصر النائبة للملفات الاصطناعية المخزنة خلف مصادقة مدعومة بالبيئة/الإعدادات | عندما يخزّن الموفّر ملفات تعريف اصطناعية نائبة لا ينبغي أن تتقدم في الأولوية                                                                  |
| 10  | `resolveDynamicModel`             | fallback متزامن لمعرّفات النماذج المملوكة للموفّر وغير الموجودة بعد في السجل المحلي   | عندما يقبل الموفّر أي معرّفات نماذج من المنبع                                                                                                 |

يتحقق كل من `normalizeModelId` و`normalizeTransport` و`normalizeConfig` أولًا من
plugin الموفّر المطابقة، ثم يمر عبر بقية provider plugins القادرة على hooks
حتى يقوم أحدها فعليًا بتغيير معرّف النموذج أو النقل/الإعدادات. وهذا يُبقي
shims الخاصة بالأسماء المستعارة/التوافق للموفّرين تعمل من دون مطالبة
المستدعي بمعرفة أي plugin مجمعة تملك إعادة الكتابة. وإذا لم تُعد أي hook
خاصة بالموفّر كتابة إدخال إعداد Google-family مدعوم، فإن مُطبِّع إعدادات Google
المجمعة لا يزال يطبق هذا التنظيف التوافقي.

إذا احتاج الموفّر إلى بروتوكول wire مخصص بالكامل أو منفذ طلبات مخصص بالكامل،
فهذا صنف مختلف من الامتداد. فهذه hooks مخصصة لسلوك الموفّر الذي
لا يزال يعمل على agent loop العادية في OpenClaw.

### مثال على موفّر

```ts
api.registerProvider({
  id: "example-proxy",
  label: "Example Proxy",
  auth: [],
  catalog: {
    order: "simple",
    run: async (ctx) => {
      const apiKey = ctx.resolveProviderApiKey("example-proxy").apiKey;
      if (!apiKey) {
        return null;
      }
      return {
        provider: {
          baseUrl: "https://proxy.example.com/v1",
          apiKey,
          api: "openai-completions",
          models: [{ id: "auto", name: "Auto" }],
        },
      };
    },
  },
  resolveDynamicModel: (ctx) => ({
    id: ctx.modelId,
    name: ctx.modelId,
    provider: "example-proxy",
    api: "openai-completions",
    baseUrl: "https://proxy.example.com/v1",
    reasoning: false,
    input: ["text"],
    cost: { input: 0, output: 0, cacheRead: 0, cacheWrite: 0 },
    contextWindow: 128000,
    maxTokens: 8192,
  }),
  prepareRuntimeAuth: async (ctx) => {
    const exchanged = await exchangeToken(ctx.apiKey);
    return {
      apiKey: exchanged.token,
      baseUrl: exchanged.baseUrl,
      expiresAt: exchanged.expiresAt,
    };
  },
  resolveUsageAuth: async (ctx) => {
    const auth = await ctx.resolveOAuthToken();
    return auth ? { token: auth.token } : null;
  },
  fetchUsageSnapshot: async (ctx) => {
    return await fetchExampleProxyUsage(ctx.token, ctx.timeoutMs, ctx.fetchFn);
  },
});
```

### أمثلة مدمجة

  التفكير الافتراضي/التكيفي لـ Claude، وتشكيل تدفق خاص بـ Anthropic من أجل
  رؤوس beta، و`/fast` / `serviceTier`، و`context1m`.

- تستخدم MiniMax وXiaomi ‏`catalog` إضافةً إلى hooks الاستخدام لأن
  سلوك `/usage` مملوك لـ plugin حتى وإن ظل الاستدلال يعمل عبر
  عمليات النقل المشتركة.

## مساعدات وقت التشغيل

يمكن للplugins الوصول إلى مجموعة مختارة من مساعدات core عبر `api.runtime`. بالنسبة إلى TTS:

```ts
const clip = await api.runtime.tts.textToSpeech({
  text: "Hello from OpenClaw",
  cfg: api.config,
});

const result = await api.runtime.tts.textToSpeechTelephony({
  text: "Hello from OpenClaw",
  cfg: api.config,
});

const voices = await api.runtime.tts.listVoices({
  provider: "elevenlabs",
  cfg: api.config,
});
```

ملاحظات:

- يعيد `textToSpeech` حمولة مخرجات TTS العادية من core لأسطح الملفات/الملاحظات الصوتية.
- يستخدم إعدادات `messages.tts` الأساسية واختيار الموفّر.
- يعيد مخزن PCM الصوتي + معدل العينة. ويجب على plugins إعادة أخذ العينات/الترميز لموفّريها.
- يكون `listVoices` اختياريًا لكل موفّر. استخدمه لمنتقيات الأصوات أو تدفقات الإعداد المملوكة للمورّد.
- يمكن أن تتضمن قوائم الأصوات بيانات وصفية أغنى مثل اللغة المحلية، والنوع، ووسوم الشخصية لمنتقيات واعية بالموفّر.
- تدعم OpenAI وElevenLabs الاتصالات الهاتفية اليوم. أما Microsoft فلا.

يمكن للplugins أيضًا تسجيل موفّري الكلام عبر `api.registerSpeechProvider(...)`.

```ts
api.registerSpeechProvider({
  id: "acme-speech",
  label: "Acme Speech",
  isConfigured: ({ config }) => Boolean(config.messages?.tts),
  synthesize: async (req) => {
    return {
      audioBuffer: Buffer.from([]),
      outputFormat: "mp3",
      fileExtension: ".mp3",
      voiceCompatible: false,
    };
  },
});
```

ملاحظات:

- أبقِ سياسة TTS وfallback وتسليم الرد داخل core.
- استخدم موفّري الكلام للسلوك التركيبي المملوك للمورّد.
- يُطبَّع الإدخال القديم لـ Microsoft ‏`edge` إلى معرّف الموفّر `microsoft`.
- نموذج الملكية المفضل موجّه بحسب الشركة: إذ يمكن لـ plugin مورّد واحدة أن
  تملك موفّرات النص، والكلام، والصور، والوسائط المستقبلية مع إضافة OpenClaw
  لعقود القدرات تلك.

بالنسبة إلى فهم الصور/الصوت/الفيديو، تسجّل plugins
موفّر فهم وسائط مكتوب النوع بدلًا من كيس مفتاح/قيمة عام:

```ts
api.registerMediaUnderstandingProvider({
  id: "google",
  capabilities: ["image", "audio", "video"],
  describeImage: async (req) => ({ text: "..." }),
  transcribeAudio: async (req) => ({ text: "..." }),
  describeVideo: async (req) => ({ text: "..." }),
});
```

ملاحظات:

- أبقِ التنسيق، وfallback، والإعدادات، وربط القنوات داخل core.
- أبقِ سلوك المورّد داخل provider plugin.
- يجب أن يبقى التوسع الإضافي مكتوب النوع: طرق اختيارية جديدة، وحقول نتائج اختيارية جديدة، وقدرات اختيارية جديدة.
- ويتبع توليد الفيديو بالفعل النمط نفسه:
  - يملك core عقد القدرة ومساعد وقت التشغيل
  - تسجّل vendor plugins ‏`api.registerVideoGenerationProvider(...)`
  - وتستهلك feature/channel plugins ‏`api.runtime.videoGeneration.*`

وبالنسبة إلى مساعدات وقت تشغيل media-understanding، يمكن للplugins استدعاء:

```ts
const image = await api.runtime.mediaUnderstanding.describeImageFile({
  filePath: "/tmp/inbound-photo.jpg",
  cfg: api.config,
  agentDir: "/tmp/agent",
});

const video = await api.runtime.mediaUnderstanding.describeVideoFile({
  filePath: "/tmp/inbound-video.mp4",
  cfg: api.config,
});
```

أما بالنسبة إلى نسخ الصوت، فيمكن للplugins استخدام وقت تشغيل media-understanding
أو الاسم المستعار الأقدم STT:

```ts
const { text } = await api.runtime.mediaUnderstanding.transcribeAudioFile({
  filePath: "/tmp/inbound-audio.ogg",
  cfg: api.config,
  // Optional when MIME cannot be inferred reliably:
  mime: "audio/ogg",
});
```

ملاحظات:

- يُعد `api.runtime.mediaUnderstanding.*` السطح المشترك المفضل لـ
  فهم الصور/الصوت/الفيديو.
- يستخدم إعدادات الصوت لفهم الوسائط في core ‏(`tools.media.audio`) وترتيب fallback للموفّرين.
- يعيد `{ text: undefined }` عندما لا يتم إنتاج مخرجات نسخ (على سبيل المثال عند التخطي/إدخال غير مدعوم).
- يبقى `api.runtime.stt.transcribeAudioFile(...)` اسمًا مستعارًا للتوافق.

يمكن للplugins أيضًا تشغيل عمليات subagent في الخلفية عبر `api.runtime.subagent`:

```ts
const result = await api.runtime.subagent.run({
  sessionKey: "agent:main:subagent:search-helper",
  message: "Expand this query into focused follow-up searches.",
  provider: "openai",
  model: "gpt-4.1-mini",
  deliver: false,
});
```

ملاحظات:

- `provider` و`model` تجاوزات اختيارية لكل تشغيل، وليستا تغييرات جلسة دائمة.
- لا يحترم OpenClaw حقول التجاوز هذه إلا للجهات المستدعية الموثوقة.
- بالنسبة إلى عمليات fallback المملوكة لـ plugin، يجب على المشغّلين الاشتراك صراحةً عبر `plugins.entries.<id>.subagent.allowModelOverride: true`.
- استخدم `plugins.entries.<id>.subagent.allowedModels` لتقييد plugins الموثوقة بأهداف `provider/model` القانونية المحددة، أو `"*"` للسماح بأي هدف صراحةً.
- لا تزال عمليات subagent الخاصة بالplugins غير الموثوقة تعمل، لكن تُرفض طلبات التجاوز بدلًا من أن تعود إلى fallback بصمت.

وبالنسبة إلى البحث في الويب، يمكن للplugins استهلاك مساعد وقت التشغيل المشترك بدلًا من
الوصول إلى ربط أداة agent:

```ts
const providers = api.runtime.webSearch.listProviders({
  config: api.config,
});

const result = await api.runtime.webSearch.search({
  config: api.config,
  args: {
    query: "OpenClaw plugin runtime helpers",
    count: 5,
  },
});
```

كما يمكن للplugins أيضًا تسجيل موفّري بحث ويب عبر
`api.registerWebSearchProvider(...)`.

ملاحظات:

- أبقِ اختيار الموفّر، وحل بيانات الاعتماد، ودلالات الطلب المشتركة داخل core.
- استخدم موفّري البحث في الويب لوسائل النقل الخاصة بالمورّد.
- يُعد `api.runtime.webSearch.*` السطح المشترك المفضل لـ feature/channel plugins التي تحتاج إلى سلوك بحث من دون الاعتماد على غلاف أداة agent.

## مسارات Gateway HTTP

يمكن للplugins كشف نقاط نهاية HTTP عبر `api.registerHttpRoute(...)`.

```ts
api.registerHttpRoute({
  path: "/acme/webhook",
  auth: "plugin",
  match: "exact",
  handler: async (_req, res) => {
    res.statusCode = 200;
    res.end("ok");
    return true;
  },
});
```

حقول المسار:

- `path`: مسار route ضمن خادم Gateway HTTP.
- `auth`: مطلوب. استخدم `"gateway"` لطلب مصادقة gateway العادية، أو `"plugin"` للمصادقة/التحقق من webhook المُدار بواسطة plugin.
- `match`: اختياري. `"exact"` (الافتراضي) أو `"prefix"`.
- `replaceExisting`: اختياري. يتيح لـ plugin نفسها استبدال تسجيل المسار الحالي الخاص بها.
- `handler`: أعد `true` عندما يتعامل route مع الطلب.

ملاحظات:

- تمت إزالة `api.registerHttpHandler(...)` وستتسبب في خطأ تحميل plugin. استخدم `api.registerHttpRoute(...)` بدلًا منها.
- يجب أن تعلن مسارات plugin عن `auth` صراحةً.
- تُرفض تعارضات `path + match` الدقيقة ما لم تكن `replaceExisting: true`، ولا يمكن لـ plugin واحدة استبدال route مملوكة لـ plugin أخرى.
- تُرفض المسارات المتداخلة ذات مستويات `auth` المختلفة. أبقِ سلاسل fallthrough بين `exact`/`prefix` على مستوى المصادقة نفسه فقط.
- لا تتلقى المسارات ذات `auth: "plugin"` نطاقات runtime الخاصة بالمشغّل تلقائيًا. فهي مخصصة لـ webhooks/التحقق من التوقيع المُدار بواسطة plugin، لا لاستدعاءات مساعدات Gateway ذات الامتيازات.
- تعمل المسارات ذات `auth: "gateway"` داخل نطاق runtime لطلب Gateway، لكن هذا النطاق محافظ عمدًا:
  - تبقي مصادقة bearer عبر السر المشترك (`gateway.auth.mode = "token"` / `"password"`) نطاقات runtime لمسارات plugin مثبتة على `operator.write`، حتى لو أرسل المستدعي `x-openclaw-scopes`
  - تحترم أوضاع HTTP الموثوقة الحاملة للهوية (مثل `trusted-proxy` أو `gateway.auth.mode = "none"` على ingress خاص) قيمة `x-openclaw-scopes` فقط عندما يكون هذا الرأس موجودًا صراحةً
  - إذا كان `x-openclaw-scopes` غائبًا في تلك الطلبات الحاملة للهوية الخاصة بمسارات plugin، يعود نطاق runtime إلى `operator.write`
- القاعدة العملية: لا تفترض أن route خاصة بـ plugin ومحمية بمصادقة gateway هي سطح إدارة ضمني. فإذا احتاج route لديك إلى سلوك خاص بالإدارة فقط، فاطلب وضع مصادقة حاملًا للهوية ووثّق عقد الرأس `x-openclaw-scopes` الصريح.

## مسارات استيراد Plugin SDK

استخدم المسارات الفرعية لـ SDK بدلًا من الاستيراد الأحادي `openclaw/plugin-sdk` عند
تأليف plugins:

- `openclaw/plugin-sdk/plugin-entry` من أجل بدائيات تسجيل plugin.
- `openclaw/plugin-sdk/core` من أجل العقدة العامة المشتركة الموجهة إلى plugin.
- `openclaw/plugin-sdk/config-schema` من أجل تصدير مخطط Zod الجذر لـ `openclaw.json`
  (`OpenClawSchema`).
- بدائيات القنوات المستقرة مثل `openclaw/plugin-sdk/channel-setup`,
  و`openclaw/plugin-sdk/setup-runtime`,
  و`openclaw/plugin-sdk/setup-adapter-runtime`,
  و`openclaw/plugin-sdk/setup-tools`,
  و`openclaw/plugin-sdk/channel-pairing`,
  و`openclaw/plugin-sdk/channel-contract`,
  و`openclaw/plugin-sdk/channel-feedback`,
  و`openclaw/plugin-sdk/channel-inbound`,
  و`openclaw/plugin-sdk/channel-lifecycle`,
  و`openclaw/plugin-sdk/channel-reply-pipeline`,
  و`openclaw/plugin-sdk/command-auth`,
  و`openclaw/plugin-sdk/secret-input`, و
  `openclaw/plugin-sdk/webhook-ingress` من أجل ربط
  الإعداد/المصادقة/الرد/webhook المشترك. ويُعد `channel-inbound` الموطن المشترك لـ debounce، ومطابقة mention، وتنسيق envelope، ومساعدات سياق envelope الواردة.
  كما أن `channel-setup` هو seam الإعداد الاختياري الضيق.
  و`setup-runtime` هو سطح الإعداد الآمن لوقت التشغيل المستخدم من قبل `setupEntry` /
  والبدء المؤجل، بما في ذلك adapters تصحيح الإعداد الآمنة للاستيراد.
  و`setup-adapter-runtime` هو seam adapter إعداد الحساب الواعية بالبيئة.
  و`setup-tools` هو seam المساعدة الصغيرة الخاصة بـ CLI/الأرشيف/الوثائق (`formatCliCommand`,
  و`detectBinary`, و`extractArchive`, و`resolveBrewExecutable`, و`formatDocsLink`,
  و`CONFIG_DIR`).
- المسارات الفرعية الخاصة بالمجال مثل `openclaw/plugin-sdk/channel-config-helpers`,
  و`openclaw/plugin-sdk/allow-from`,
  و`openclaw/plugin-sdk/channel-config-schema`,
  و`openclaw/plugin-sdk/telegram-command-config`,
  و`openclaw/plugin-sdk/channel-policy`,
  و`openclaw/plugin-sdk/approval-runtime`,
  و`openclaw/plugin-sdk/config-runtime`,
  و`openclaw/plugin-sdk/infra-runtime`,
  و`openclaw/plugin-sdk/agent-runtime`,
  و`openclaw/plugin-sdk/lazy-runtime`,
  و`openclaw/plugin-sdk/reply-history`,
  و`openclaw/plugin-sdk/routing`,
  و`openclaw/plugin-sdk/status-helpers`,
  و`openclaw/plugin-sdk/text-runtime`,
  و`openclaw/plugin-sdk/runtime-store`, و
  `openclaw/plugin-sdk/directory-runtime` من أجل مساعدات وقت التشغيل/الإعدادات المشتركة.
  ويُعد `telegram-command-config` seam عامًا ضيقًا من أجل
  تطبيع/التحقق من الأوامر المخصصة في Telegram ويبقى متاحًا حتى لو كان سطح العقد المجمّع لـ Telegram غير متاح مؤقتًا.
  أما `text-runtime` فهو seam النص/Markdown/التسجيلات المشتركة، بما في ذلك
  إزالة النص المرئي للمساعد، ومساعدات عرض/تجزئة markdown، ومساعدات الإخفاء،
  ومساعدات وسم التوجيه، وأدوات النص الآمن.
- ينبغي أن تفضّل seams الخاصة بالموافقات في القنوات عقد
  `approvalCapability` واحدًا على plugin. ثم يقرأ core بعد ذلك مصادقة الموافقة، والتسليم، والعرض، وسلوك التوجيه الأصلي من خلال هذه القدرة الواحدة بدلًا من خلط
  سلوك الموافقة في حقول plugin غير ذات الصلة.
- تُعد `openclaw/plugin-sdk/channel-runtime` مهجورة وتبقى فقط بوصفها shim للتوافق مع plugins الأقدم. ويجب أن تستورد الشيفرة الجديدة البدائيات العامة الأضيق بدلًا منها، كما يجب ألا تضيف شيفرة المستودع استيرادات جديدة إلى shim هذه.
- تبقى الجوانب الداخلية للامتدادات المجمعة خاصة. وينبغي للplugins الخارجية أن تستخدم فقط المسارات الفرعية `openclaw/plugin-sdk/*`. ويمكن لشيفرة core/test في OpenClaw استخدام نقاط الدخول العامة في المستودع ضمن جذر حزمة plugin مثل `index.js`, و`api.js`, و`runtime-api.js`, و`setup-entry.js`، والملفات الضيقة النطاق مثل `login-qr-api.js`. ولا تستورد أبدًا `src/*` الخاصة بحزمة plugin من core أو من امتداد آخر.
- تقسيم نقطة دخول المستودع:
  `<plugin-package-root>/api.js` هو شريط المساعدات/الأنواع،
  و`<plugin-package-root>/runtime-api.js` هو شريط وقت التشغيل فقط،
  و`<plugin-package-root>/index.js` هو entry الخاصة بـ plugin المجمعة،
  و`<plugin-package-root>/setup-entry.js` هو entry الخاصة بـ setup plugin.
- أمثلة موفّرين مجمعة حالية:
  - تستخدم Anthropic ‏`api.js` / `contract-api.js` لمساعدات تدفق Claude مثل
    `wrapAnthropicProviderStream`, ومساعدات رؤوس beta، وتحليل `service_tier`.
  - تستخدم OpenAI ‏`api.js` لبناة الموفّرين، ومساعدات النموذج الافتراضي،
    وبناة الموفّرين الفوريين.
  - تستخدم OpenRouter ‏`api.js` لباني الموفّر إضافةً إلى مساعدات
    التهيئة/الإعداد، بينما يمكن لـ `register.runtime.js` أن يعيد تصدير
    مساعدات `plugin-sdk/provider-stream` العامة للاستخدام المحلي داخل المستودع.
- تفضّل نقاط الدخول العامة المحملة عبر facade لقطة إعدادات runtime النشطة
  عندما تكون موجودة، ثم تعود إلى ملف الإعدادات المحلول على القرص عندما
  لا يكون OpenClaw قد بدأ بعد في تقديم لقطة runtime.
- تبقى البدائيات العامة المشتركة العقد المفضل في SDK. وما تزال مجموعة صغيرة
  محجوزة من seams التوافقية لمساعدات تحمل علامات القنوات المجمعة موجودة. تعامل مع هذه
  بوصفها seams للصيانة/التوافق مع الحزم المجمعة، لا أهداف استيراد جديدة
  لأطراف ثالثة؛ إذ ينبغي أن تستقر العقود الجديدة المشتركة بين القنوات على
  المسارات العامة `plugin-sdk/*` أو على أشرطة `api.js` /
  `runtime-api.js` المحلية الخاصة بـ plugin.

ملاحظة توافق:

- تجنب الشريط الجذري `openclaw/plugin-sdk` في الشيفرة الجديدة.
- فضّل أولًا البدائيات المستقرة الضيقة. فالمسارات الفرعية الأحدث الخاصة بالإعداد/الاقتران/الرد/
  والتغذية الراجعة/العقد/الوارد/الخيوط/الأوامر/secret-input/webhook/infra/
  وقائمة السماح/الحالة/message-tool هي العقد المقصود للعمل الجديد في
  plugins المجمعة والخارجية.
  ويجب أن ينتمي تحليل/مطابقة الأهداف إلى `openclaw/plugin-sdk/channel-targets`.
  أما بوابات message action ومساعدات message-id الخاصة بالتفاعلات فتنتمي إلى
  `openclaw/plugin-sdk/channel-actions`.
- ليست أشرطة المساعدات الخاصة بالامتدادات المجمعة مستقرة افتراضيًا. فإذا
  كانت المساعدة مطلوبة فقط لامتداد مجمّع، فأبقِها خلف seam المحلي
  `api.js` أو `runtime-api.js` لذلك الامتداد بدلًا من ترقيتها إلى
  `openclaw/plugin-sdk/<extension>`.
- يجب أن تكون seams المساعدات المشتركة الجديدة عامة، لا موسومة باسم قناة. فتحليل
  الأهداف المشترك ينتمي إلى `openclaw/plugin-sdk/channel-targets`؛ أما
  الجوانب الداخلية الخاصة بالقنوات فتبقى خلف `api.js` أو `runtime-api.js`
  المحلية للplugin المالكة.
- توجد مسارات فرعية خاصة بالقدرات مثل `image-generation`,
  و`media-understanding`, و`speech` لأن plugins المجمعة/الأصلية تستخدمها
  اليوم. لكن وجودها لا يعني بحد ذاته أن كل مساعدة مُصدرة فيها
  عقد خارجي مجمّد طويل الأمد.

## مخططات أداة الرسائل

ينبغي أن تملك plugins مساهمات المخطط الخاصة بالقناة في `describeMessageTool(...)`.
وأبقِ الحقول الخاصة بالمورّد داخل plugin، لا في core المشترك.

أما بالنسبة إلى أجزاء المخطط المحمولة المشتركة، فأعد استخدام المساعدات العامة المصدّرة عبر
`openclaw/plugin-sdk/channel-actions`:

- `createMessageToolButtonsSchema()` من أجل حمولات بأسلوب شبكة الأزرار
- `createMessageToolCardSchema()` من أجل حمولات البطاقات المهيكلة

إذا كان شكل مخطط ما لا يكون منطقيًا إلا لموفّر واحد، فعرّفه في مصدر
ذلك plugin نفسه بدلًا من ترقيته إلى SDK المشتركة.

## حل أهداف القنوات

ينبغي أن تملك channel plugins دلالات الأهداف الخاصة بالقناة. وأبقِ مضيف
الإرسال الخارجي عامًا واستخدم سطح messaging adapter لقواعد الموفّر:

- يقرر `messaging.inferTargetChatType({ to })` ما إذا كان يجب التعامل مع هدف
  مطبّع على أنه `direct` أو `group` أو `channel` قبل البحث في الدليل.
- يحدد `messaging.targetResolver.looksLikeId(raw, normalized)` ما إذا كان يجب على core
  تخطي البحث مباشرةً إلى حل شبيه بالمعرّف بدلًا من البحث في الدليل.
- `messaging.targetResolver.resolveTarget(...)` هو fallback الخاص بـ plugin عندما
  يحتاج core إلى حل نهائي يملكه الموفّر بعد التطبيع أو بعد فشل
  البحث في الدليل.
- تملك `messaging.resolveOutboundSessionRoute(...)` إنشاء
  مسار الجلسة الصادر الخاص بالموفّر بمجرد حل الهدف.

التقسيم الموصى به:

- استخدم `inferTargetChatType` لقرارات الفئة التي يجب أن تحدث قبل
  البحث في الزملاء/المجموعات.
- استخدم `looksLikeId` لفحوصات "تعامل مع هذا على أنه معرّف هدف
  أصلي/صريح".
- استخدم `resolveTarget` من أجل fallback التطبيع الخاص بالموفّر، لا من أجل
  البحث العام في الدليل.
- أبقِ المعرّفات الأصلية الخاصة بالموفّر مثل chat ids، وthread ids، وJIDs، وhandles، وroom ids داخل قيم `target` أو
  المعلمات الخاصة بالموفّر، لا في حقول SDK العامة.

## الأدلة المدعومة بالإعدادات

ينبغي أن تحتفظ plugins التي تشتق إدخالات الدليل من الإعدادات
بهذا المنطق داخل plugin وأن تعيد استخدام المساعدات المشتركة من
`openclaw/plugin-sdk/directory-runtime`.

استخدم هذا عندما تحتاج قناة إلى زملاء/مجموعات مدعومين بالإعدادات مثل:

- زملاء الرسائل المباشرة المدفوعين بقائمة السماح
- خرائط القنوات/المجموعات المضبوطة
- fallback ثابتة للدليل على مستوى الحساب

ولا تتعامل المساعدات المشتركة في `directory-runtime` إلا مع العمليات العامة:

- تصفية الاستعلام
- تطبيق الحد
- مساعدات إزالة التكرار/التطبيع
- بناء `ChannelDirectoryEntry[]`

أما فحص الحساب الخاص بالقناة وتطبيع المعرّفات فينبغي أن يبقيا داخل
تنفيذ plugin.

## كتالوجات الموفّرين

يمكن لـ provider plugins تعريف كتالوجات نماذج للاستدلال باستخدام
`registerProvider({ catalog: { run(...) { ... } } })`.

تعيد `catalog.run(...)` الشكل نفسه الذي يكتبه OpenClaw في
`models.providers`:

- `{ provider }` لإدخال موفّر واحد
- `{ providers }` لإدخالات موفّرين متعددة

استخدم `catalog` عندما تملك plugin معرّفات نماذج خاصة بالموفّر، أو افتراضيات base URL، أو بيانات وصفية للنماذج مقيدة بالمصادقة.

يتحكم `catalog.order` في توقيت دمج كتالوج plugin بالنسبة إلى
الموفّرين الضمنيين المدمجين في OpenClaw:

- `simple`: موفّرون عاديون مدفوعون بمفتاح API أو البيئة
- `profile`: موفّرون يظهرون عند وجود auth profiles
- `paired`: موفّرون يصطنعون إدخالات متعددة ذات صلة
- `late`: التمريرة الأخيرة، بعد الموفّرين الضمنيين الآخرين

تفوز الموفّرات اللاحقة عند تصادم المفاتيح، لذلك يمكن للplugins عمدًا تجاوز إدخال
موفّر مدمج يحمل معرّف الموفّر نفسه.

التوافق:

- لا يزال `discovery` يعمل بوصفه اسمًا مستعارًا قديمًا
- إذا تم تسجيل `catalog` و`discovery` معًا، يستخدم OpenClaw ‏`catalog`

## فحص القناة للقراءة فقط

إذا كانت plugin لديك تسجّل قناة، ففضّل تنفيذ
`plugin.config.inspectAccount(cfg, accountId)` إلى جانب `resolveAccount(...)`.

السبب:

- `resolveAccount(...)` هو مسار وقت التشغيل. ويُسمح له بافتراض أن بيانات الاعتماد
  قد تجسدت بالكامل وأن يفشل سريعًا عند غياب الأسرار المطلوبة.
- لا ينبغي للمسارات الخاصة بالأوامر المخصصة للقراءة فقط مثل `openclaw status`, و`openclaw status --all`,
  و`openclaw channels status`, و`openclaw channels resolve`, وتدفقات doctor/config repair
  أن تحتاج إلى تجسيد بيانات اعتماد وقت التشغيل لمجرد وصف
  الإعدادات.

السلوك الموصى به لـ `inspectAccount(...)`:

- أعد فقط حالة الحساب الوصفية.
- حافظ على `enabled` و`configured`.
- ضمّن حقول مصدر/حالة بيانات الاعتماد عند الاقتضاء، مثل:
  - `tokenSource`, `tokenStatus`
  - `botTokenSource`, `botTokenStatus`
  - `appTokenSource`, `appTokenStatus`
  - `signingSecretSource`, `signingSecretStatus`
- لا تحتاج إلى إعادة قيم الرموز الخام لمجرد الإبلاغ عن
  التوفر في وضع القراءة فقط. يكفي إرجاع `tokenStatus: "available"` (ومعه حقل المصدر المطابق) لأوامر شبيهة بـ status.
- استخدم `configured_unavailable` عندما تكون بيانات الاعتماد مضبوطة عبر SecretRef ولكنها
  غير متاحة في مسار الأمر الحالي.

وهذا يتيح للأوامر المخصصة للقراءة فقط الإبلاغ عن "مُعدّ لكنه غير متاح في
مسار الأمر هذا" بدلًا من التعطل أو الإبلاغ الخاطئ عن أن الحساب غير مُعدّ.

## حزم Package packs

يمكن أن يتضمن دليل plugin ملف `package.json` يحوي `openclaw.extensions`:

```json
{
  "name": "my-pack",
  "openclaw": {
    "extensions": ["./src/safety.ts", "./src/tools.ts"],
    "setupEntry": "./src/setup-entry.ts"
  }
}
```

يصبح كل entry plugin. وإذا كانت الحزمة تسرد امتدادات متعددة، يصبح معرّف plugin
`name/<fileBase>`.

إذا كانت plugin لديك تستورد تبعيات npm، فثبّتها في ذلك الدليل حتى
يتوفر `node_modules` (`npm install` / `pnpm install`).

بوابة أمان: يجب أن يبقى كل entry في `openclaw.extensions` داخل دليل plugin
بعد حل symlink. وتُرفض الإدخالات التي تهرب خارج دليل package.

ملاحظة أمان: يقوم `openclaw plugins install` بتثبيت تبعيات plugin عبر
`npm install --omit=dev --ignore-scripts` (من دون lifecycle scripts، ومن دون تبعيات تطوير وقت التشغيل). أبقِ أشجار تبعيات plugin "JavaScript/TypeScript خالصة" وتجنب الحزم التي تتطلب بناء `postinstall`.

اختياري: يمكن لـ `openclaw.setupEntry` أن تشير إلى module خفيفة مخصصة للإعداد فقط.
وعندما يحتاج OpenClaw إلى أسطح الإعداد الخاصة بـ channel plugin معطلة، أو
عندما تكون channel plugin مفعّلة لكنها لا تزال غير مُعدّة، فإنه يحمّل `setupEntry`
بدلًا من entry الكاملة للplugin. وهذا يبقي بدء التشغيل والإعداد أخف
عندما تكون entry الرئيسية في plugin لديك تربط أيضًا أدوات، أو hooks، أو غيرها من
شيفرات وقت التشغيل فقط.

اختياري: يمكن لـ `openclaw.startup.deferConfiguredChannelFullLoadUntilAfterListen`
أن تجعل channel plugin تستخدم المسار نفسه `setupEntry` خلال
مرحلة بدء التشغيل السابقة للاستماع في gateway، حتى عندما تكون القناة مُعدة بالفعل.

استخدم هذا فقط عندما يغطي `setupEntry` سطح بدء التشغيل بالكامل الذي يجب أن
يوجد قبل أن تبدأ gateway بالاستماع. وعمليًا، يعني ذلك أن entry الإعداد
يجب أن تسجّل كل قدرة مملوكة للقناة يعتمد عليها بدء التشغيل، مثل:

- تسجيل القناة نفسها
- أي مسارات HTTP يجب أن تكون متاحة قبل أن تبدأ gateway بالاستماع
- أي طرق Gateway أو أدوات أو خدمات يجب أن توجد خلال النافذة نفسها

إذا كانت entry الكاملة لديك لا تزال تملك أي قدرة مطلوبة عند بدء التشغيل، فلا تمكّن
هذا العلم. وأبقِ plugin على السلوك الافتراضي ودع OpenClaw يحمّل
entry الكاملة أثناء بدء التشغيل.

كما يمكن للقنوات المجمعة أن تنشر مساعدات على سطح عقد للإعداد فقط يمكن لـ core
الرجوع إليها قبل تحميل وقت تشغيل القناة الكامل. وسطح الترقية الحالي للإعداد هو:

- `singleAccountKeysToMove`
- `namedAccountPromotionKeys`
- `resolveSingleAccountPromotionTarget(...)`

ويستخدم Core هذا السطح عندما يحتاج إلى ترقية إعداد قناة قديمة أحادية الحساب إلى
`channels.<id>.accounts.*` من دون تحميل entry الكاملة للplugin.
وتُعد Matrix المثال المجمّع الحالي: فهي تنقل فقط مفاتيح
المصادقة/التمهيد إلى حساب مسمى مُرقّى عندما تكون الحسابات المسماة موجودة بالفعل، ويمكنها
الحفاظ على مفتاح حساب افتراضي غير قانوني مُعدّ بدلًا من إنشاء
`accounts.default` دائمًا.

وتُبقي adapters تصحيح الإعداد تلك اكتشاف سطح العقد المجمّعة كسولًا. إذ يظل وقت الاستيراد خفيفًا؛ ويتم تحميل سطح الترقية فقط عند أول استخدام بدلًا من إعادة دخول بدء تشغيل القناة المجمعة عند استيراد module.

وعندما تتضمن أسطح بدء التشغيل تلك طرق Gateway RPC، فأبقها على
بادئة خاصة بـ plugin. إذ تبقى مساحات أسماء الإدارة في core (`config.*`,
`exec.approvals.*`, `wizard.*`, `update.*`) محجوزة وتُحل دائمًا إلى `operator.admin`,
حتى لو طلبت plugin نطاقًا أضيق.

مثال:

```json
{
  "name": "@scope/my-channel",
  "openclaw": {
    "extensions": ["./index.ts"],
    "setupEntry": "./setup-entry.ts",
    "startup": {
      "deferConfiguredChannelFullLoadUntilAfterListen": true
    }
  }
}
```

### البيانات الوصفية لكتالوج القنوات

يمكن لـ channel plugins الإعلان عن بيانات إعداد/اكتشاف وصفية عبر `openclaw.channel` و
تلميحات التثبيت عبر `openclaw.install`. وهذا يُبقي core خاليًا من بيانات الكتالوج.

مثال:

```json
{
  "name": "@openclaw/nextcloud-talk",
  "openclaw": {
    "extensions": ["./index.ts"],
    "channel": {
      "id": "nextcloud-talk",
      "label": "Nextcloud Talk",
      "selectionLabel": "Nextcloud Talk (self-hosted)",
      "docsPath": "/channels/nextcloud-talk",
      "docsLabel": "nextcloud-talk",
      "blurb": "Self-hosted chat via Nextcloud Talk webhook bots.",
      "order": 65,
      "aliases": ["nc-talk", "nc"]
    },
    "install": {
      "npmSpec": "@openclaw/nextcloud-talk",
      "localPath": "<bundled-plugin-local-path>",
      "defaultChoice": "npm"
    }
  }
}
```

حقول `openclaw.channel` المفيدة إلى جانب المثال الأدنى:

- `detailLabel`: تسمية ثانوية لأسطح كتالوج/حالة أغنى
- `docsLabel`: تجاوز نص الرابط لرابط الوثائق
- `preferOver`: معرّفات plugins/قنوات أقل أولوية ينبغي أن يتفوق عليها إدخال الكتالوج هذا
- `selectionDocsPrefix`, و`selectionDocsOmitLabel`, و`selectionExtras`: عناصر تحكم النسخ الخاصة بسطح الاختيار
- `markdownCapable`: يحدد القناة على أنها قادرة على Markdown من أجل قرارات التنسيق الصادر
- `exposure.configured`: إخفاء القناة من أسطح عرض القنوات المُعدة عندما تُضبط على `false`
- `exposure.setup`: إخفاء القناة من منتقيات الإعداد/التهيئة التفاعلية عندما تُضبط على `false`
- `exposure.docs`: وسم القناة على أنها داخلية/خاصة من أجل أسطح التنقل في الوثائق
- `showConfigured` / `showInSetup`: أسماء مستعارة قديمة ما تزال مقبولة للتوافق؛ فضّل `exposure`
- `quickstartAllowFrom`: إدخال القناة في تدفق `allowFrom` القياسي الخاص بالبداية السريعة
- `forceAccountBinding`: فرض ربط صريح للحساب حتى عندما يوجد حساب واحد فقط
- `preferSessionLookupForAnnounceTarget`: تفضيل بحث الجلسة عند حل أهداف الإعلان

يمكن لـ OpenClaw أيضًا دمج **كتالوجات قنوات خارجية** (على سبيل المثال تصدير سجل
MPM). ضع ملف JSON في أحد المواقع التالية:

- `~/.openclaw/mpm/plugins.json`
- `~/.openclaw/mpm/catalog.json`
- `~/.openclaw/plugins/catalog.json`

أو وجّه `OPENCLAW_PLUGIN_CATALOG_PATHS` (أو `OPENCLAW_MPM_CATALOG_PATHS`) إلى
ملف JSON واحد أو أكثر (مفصولة بفاصلة/فاصلة منقوطة/`PATH`). ويجب أن يحتوي كل ملف
على `{ "entries": [ { "name": "@scope/pkg", "openclaw": { "channel": {...}, "install": {...} } } ] }`. كما يقبل المحلل `"packages"` أو `"plugins"` بوصفهما اسمين مستعارين قديمين لمفتاح `"entries"`.

## Context engine plugins

تملك Context engine plugins تنسيق سياق الجلسة من أجل الإدخال، والتجميع،
والضغط. سجّلها من plugin لديك باستخدام
`api.registerContextEngine(id, factory)`، ثم اختر المحرك النشط عبر
`plugins.slots.contextEngine`.

استخدم هذا عندما تحتاج plugin لديك إلى استبدال مسار السياق الافتراضي أو
توسيعه بدلًا من مجرد إضافة بحث الذاكرة أو hooks.

```ts
export default function (api) {
  api.registerContextEngine("lossless-claw", () => ({
    info: { id: "lossless-claw", name: "Lossless Claw", ownsCompaction: true },
    async ingest() {
      return { ingested: true };
    },
    async assemble({ messages }) {
      return { messages, estimatedTokens: 0 };
    },
    async compact() {
      return { ok: true, compacted: false };
    },
  }));
}
```

إذا كان محركك **لا** يملك خوارزمية الضغط، فأبقِ `compact()`
منفذة وفوّضها صراحةً:

```ts
import { delegateCompactionToRuntime } from "openclaw/plugin-sdk/core";

export default function (api) {
  api.registerContextEngine("my-memory-engine", () => ({
    info: {
      id: "my-memory-engine",
      name: "My Memory Engine",
      ownsCompaction: false,
    },
    async ingest() {
      return { ingested: true };
    },
    async assemble({ messages }) {
      return { messages, estimatedTokens: 0 };
    },
    async compact(params) {
      return await delegateCompactionToRuntime(params);
    },
  }));
}
```

## إضافة قدرة جديدة

عندما تحتاج plugin إلى سلوك لا يناسب API الحالية، فلا تتجاوز
نظام plugin عبر اختراق خاص. بل أضف القدرة المفقودة.

التسلسل الموصى به:

1. تعريف عقد core
   قرر ما السلوك المشترك الذي يجب أن يملكه core: السياسة، وfallback، ودمج الإعدادات،
   ودورة الحياة، والدلالات الموجهة إلى القنوات، وشكل مساعد وقت التشغيل.
2. إضافة أسطح تسجيل/وقت تشغيل مكتوبة النوع خاصة بـ plugin
   وسّع `OpenClawPluginApi` و/أو `api.runtime` بأصغر سطح قدرة مفيد
   مكتوب النوع.
3. ربط مستهلكي core + channel/feature
   ينبغي للقنوات وfeature plugins أن تستهلك القدرة الجديدة عبر core،
   لا عبر استيراد تنفيذ مورّد مباشرةً.
4. تسجيل تنفيذات المورّدين
   ثم تسجّل vendor plugins واجهاتها الخلفية مقابل القدرة.
5. إضافة تغطية العقد
   أضف اختبارات حتى تبقى الملكية وشكل التسجيل صريحين مع مرور الوقت.

وهكذا يبقى OpenClaw حازمًا من دون أن يصبح مرمّزًا وفق
رؤية موفّر واحد للعالم. راجع [Capability Cookbook](/ar/plugins/architecture)
للحصول على قائمة ملفات عملية ومثال مفصل.

### قائمة التحقق الخاصة بالقدرة

عندما تضيف قدرة جديدة، ينبغي أن يمس التنفيذ عادةً هذه
الأسطح معًا:

- أنواع عقد core في `src/<capability>/types.ts`
- مشغّل core / مساعد وقت التشغيل في `src/<capability>/runtime.ts`
- سطح تسجيل API الخاصة بـ plugin في `src/plugins/types.ts`
- ربط سجل plugin في `src/plugins/registry.ts`
- كشف وقت تشغيل plugin في `src/plugins/runtime/*` عندما تحتاج
  feature/channel plugins إلى استهلاكه
- مساعدات الالتقاط/الاختبار في `src/test-utils/plugin-registration.ts`
- تأكيدات الملكية/العقد في `src/plugins/contracts/registry.ts`
- وثائق المشغّل/plugin في `docs/`

إذا كان أحد هذه الأسطح مفقودًا، فهذه عادةً علامة على أن القدرة
لم تُدمج بالكامل بعد.

### قالب القدرة

النمط الأدنى:

```ts
// core contract
export type VideoGenerationProviderPlugin = {
  id: string;
  label: string;
  generateVideo: (req: VideoGenerationRequest) => Promise<VideoGenerationResult>;
};

// plugin API
api.registerVideoGenerationProvider({
  id: "openai",
  label: "OpenAI",
  async generateVideo(req) {
    return await generateOpenAiVideo(req);
  },
});

// shared runtime helper for feature/channel plugins
const clip = await api.runtime.videoGeneration.generate({
  prompt: "Show the robot walking through the lab.",
  cfg,
});
```

نمط اختبار العقد:

```ts
expect(findVideoGenerationProviderIdsForPlugin("openai")).toEqual(["openai"]);
```

وهذا يُبقي القاعدة بسيطة:

- يملك core عقد القدرة + التنسيق
- تملك vendor plugins تنفيذات المورّد
- تستهلك feature/channel plugins مساعدات وقت التشغيل
- تُبقي اختبارات العقد الملكية صريحة
