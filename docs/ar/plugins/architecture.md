## نموذج القدرات العام

القدرات هي نموذج **الإضافة الأصلية** العام داخل OpenClaw. تسجل كل
إضافة OpenClaw أصلية مقابل نوع قدرة واحد أو أكثر:

| القدرة                | طريقة التسجيل                                 | أمثلة على الإضافات                   |
| --------------------- | --------------------------------------------- | ------------------------------------ |
| استدلال النص          | `api.registerProvider(...)`                   | `openai`, `anthropic`                |
| خلفية استدلال CLI     | `api.registerCliBackend(...)`                 | `openai`, `anthropic`                |
| الكلام                | `api.registerSpeechProvider(...)`             | `elevenlabs`, `microsoft`            |
| النسخ الفوري          | `api.registerRealtimeTranscriptionProvider(...)` | `openai`                          |
| الصوت الفوري          | `api.registerRealtimeVoiceProvider(...)`      | `openai`                             |
| فهم الوسائط           | `api.registerMediaUnderstandingProvider(...)` | `openai`, `google`                   |
| توليد الصور           | `api.registerImageGenerationProvider(...)`    | `openai`, `google`, `fal`, `minimax` |
| توليد الموسيقى        | `api.registerMusicGenerationProvider(...)`    | `google`, `minimax`                  |
| توليد الفيديو         | `api.registerVideoGenerationProvider(...)`    | `qwen`                               |
| جلب الويب             | `api.registerWebFetchProvider(...)`           | `firecrawl`                          |
| البحث على الويب       | `api.registerWebSearchProvider(...)`          | `google`                             |
| القناة / المراسلة     | `api.registerChannel(...)`                    | `msteams`, `matrix`                  |

الإضافة التي تسجل صفرًا من القدرات لكنها توفّر خطافات أو أدوات أو
خدمات هي إضافة **قديمة تعتمد على الخطافات فقط**. لا يزال هذا النمط مدعومًا بالكامل.

### موقف التوافق الخارجي

تم اعتماد نموذج القدرات في النواة ويُستخدم اليوم بواسطة الإضافات
المضمّنة/الأصلية، لكن توافق الإضافات الخارجية لا يزال يحتاج إلى معيار أدق من
"إنه مُصدَّر، إذن فهو مجمّد."

الإرشادات الحالية:

- **الإضافات الخارجية الحالية:** أبقِ عمليات التكامل المعتمدة على الخطافات تعمل؛
  واعتبر ذلك خط الأساس للتوافق
- **الإضافات المضمّنة/الأصلية الجديدة:** فضّل تسجيل القدرات الصريح بدلًا من
  التوغلات الخاصة بالمورّد أو تصميمات الخطافات فقط الجديدة
- **الإضافات الخارجية التي تتبنى تسجيل القدرات:** هذا مسموح، لكن تعامل مع
  أسطح المساعدة الخاصة بالقدرات على أنها تتطور ما لم تضع الوثائق علامة
  صريحة على أن العقد مستقر

القاعدة العملية:

- واجهات تسجيل القدرات هي الاتجاه المقصود
- تظل الخطافات القديمة هي المسار الأكثر أمانًا لعدم كسر الإضافات الخارجية أثناء
  الانتقال
- ليست كل المسارات الفرعية المصدّرة للمساعدات متساوية؛ فضّل العقد الضيق
  الموثق، وليس تصديرات المساعدة العرضية

### أشكال الإضافات

يصنّف OpenClaw كل إضافة محمّلة إلى شكل استنادًا إلى سلوك التسجيل الفعلي لها
(وليس فقط البيانات الوصفية الثابتة):

- **plain-capability** -- تسجل نوع قدرة واحدًا فقط (على سبيل المثال
  إضافة مزوّد فقط مثل `mistral`)
- **hybrid-capability** -- تسجل عدة أنواع من القدرات (على سبيل المثال
  `openai` يملك استدلال النص، والكلام، وفهم الوسائط، وتوليد
  الصور)
- **hook-only** -- تسجل خطافات فقط (مكتوبة الأنواع أو مخصصة)، بلا قدرات
  أو أدوات أو أوامر أو خدمات
- **non-capability** -- تسجل أدوات أو أوامر أو خدمات أو مسارات لكن بلا
  قدرات

استخدم `openclaw plugins inspect <id>` لرؤية شكل الإضافة وتفصيل
القدرات. راجع [مرجع CLI](/cli/plugins#inspect) للتفاصيل.

### الخطافات القديمة

لا يزال الخطاف `before_agent_start` مدعومًا كمسار توافق للإضافات
المعتمدة على الخطافات فقط. لا تزال إضافات قديمة مستخدمة فعليًا تعتمد عليه.

الاتجاه:

- أبقه يعمل
- وثّقه على أنه قديم
- فضّل `before_model_resolve` لأعمال تجاوز النموذج/المزوّد
- فضّل `before_prompt_build` لأعمال تعديل الموجّه
- أزِله فقط بعد انخفاض الاستخدام الفعلي وتأكيد تغطية الاختبارات الثابتة لسلامة الترحيل

### إشارات التوافق

عند تشغيل `openclaw doctor` أو `openclaw plugins inspect <id>`، قد ترى
إحدى هذه العلامات:

| الإشارة                    | المعنى                                                       |
| -------------------------- | ------------------------------------------------------------ |
| **config valid**           | يجري تحليل الإعدادات جيدًا وتُحل الإضافات                   |
| **compatibility advisory** | تستخدم الإضافة نمطًا مدعومًا لكنه أقدم (مثل `hook-only`)    |
| **legacy warning**         | تستخدم الإضافة `before_agent_start`، وهو مهمل               |
| **hard error**             | الإعدادات غير صالحة أو فشل تحميل الإضافة                    |

لن يؤدي `hook-only` ولا `before_agent_start` إلى كسر إضافتك اليوم --
فـ`hook-only` مجرد تنبيه إرشادي، و`before_agent_start` لا يطلق إلا تحذيرًا. تظهر هذه
الإشارات أيضًا في `openclaw status --all` و`openclaw plugins doctor`.

## نظرة عامة على البنية

يحتوي نظام الإضافات في OpenClaw على أربع طبقات:

1. **البيان + الاكتشاف**
   يعثر OpenClaw على الإضافات المرشحة من المسارات المهيأة، وجذور مساحات العمل،
   وجذور الامتدادات العامة، والامتدادات المضمّنة. يقرأ الاكتشاف بيانات
   `openclaw.plugin.json` الأصلية بالإضافة إلى بيانات الحزم المتوافقة أولًا.
2. **التمكين + التحقق**
   تقرر النواة ما إذا كانت الإضافة المكتشفة مفعّلة، أو معطّلة، أو محظورة، أو
   مختارة لمنفذ حصري مثل الذاكرة.
3. **التحميل وقت التشغيل**
   تُحمَّل إضافات OpenClaw الأصلية داخل العملية عبر jiti وتسجّل
   القدرات في سجل مركزي. وتُطبّع الحزم المتوافقة إلى
   سجلات في السجل دون استيراد كود وقت التشغيل.
4. **استهلاك الأسطح**
   يقرأ باقي OpenClaw السجل لكشف الأدوات، والقنوات، وإعداد
   المزوّد، والخطافات، ومسارات HTTP، وأوامر CLI، والخدمات.

وبالنسبة إلى CLI الخاص بالإضافة تحديدًا، يُقسَّم اكتشاف الأمر الجذري إلى مرحلتين:

- تأتي بيانات وقت التحليل من `registerCli(..., { descriptors: [...] })`
- يمكن أن يبقى نموذج CLI الحقيقي للإضافة كسولًا ويُسجَّل عند أول استدعاء

يبقي ذلك كود CLI المملوك للإضافة داخل الإضافة مع السماح لـ OpenClaw
بحجز أسماء الأوامر الجذرية قبل التحليل.

الحد التصميمي المهم:

- يجب أن يعمل الاكتشاف والتحقق من الإعدادات اعتمادًا على **بيانات البيان/المخطط**
  دون تنفيذ كود الإضافة
- يأتي سلوك وقت التشغيل الأصلي من مسار `register(api)` في وحدة الإضافة

يسمح هذا الفصل لـ OpenClaw بالتحقق من الإعدادات، وشرح الإضافات المفقودة/المعطلة،
وبناء تلميحات واجهة المستخدم/المخطط قبل تنشيط وقت التشغيل الكامل.

### إضافات القنوات وأداة الرسائل المشتركة

لا تحتاج إضافات القنوات إلى تسجيل أداة إرسال/تعديل/تفاعل منفصلة
لإجراءات الدردشة العادية. يحتفظ OpenClaw بأداة `message` مشتركة واحدة في النواة،
وتملك إضافات القنوات الاكتشاف والتنفيذ الخاصين بالقناة وراءها.

الحد الحالي هو:

- تملك النواة مضيف أداة `message` المشتركة، وربط الموجّه، ومسك
  مفاتيح الجلسة/السلسلة، وتوجيه التنفيذ
- تملك إضافات القنوات اكتشاف الإجراءات المقيّدة، واكتشاف القدرات، وأي
  أجزاء مخطط خاصة بالقناة
- تملك إضافات القنوات قواعد المحادثة الخاصة بالجلسة بحسب المزوّد، مثل
  كيفية ترميز معرّفات المحادثة لمعرّفات السلاسل أو وراثتها من المحادثات الأصلية
- تنفذ إضافات القنوات الإجراء النهائي عبر مهايئ الإجراءات الخاص بها

بالنسبة إلى إضافات القنوات، سطح SDK هو
`ChannelMessageActionAdapter.describeMessageTool(...)`. يتيح هذا الاستدعاء
الموحّد للاكتشاف للإضافة أن تعيد إجراءاتها المرئية، وقدراتها، ومساهماتها
في المخطط معًا حتى لا تنفصل هذه الأجزاء عن بعضها.

تمرر النواة نطاق وقت التشغيل إلى خطوة الاكتشاف هذه. وتشمل الحقول المهمة:

- `accountId`
- `currentChannelId`
- `currentThreadTs`
- `currentMessageId`
- `sessionKey`
- `sessionId`
- `agentId`
- `requesterSenderId` الوارد الموثوق

وهذا مهم للإضافات الحساسة للسياق. يمكن للقناة إخفاء أو إظهار
إجراءات الرسائل استنادًا إلى الحساب النشط، أو الغرفة/السلسلة/الرسالة الحالية، أو
هوية الطالب الموثوقة، دون ترميز فروع خاصة بالقناة داخل أداة
`message` في النواة.

ولهذا السبب لا تزال تغييرات توجيه العداء المضمّن من عمل الإضافة: إذ يقع على العداء
مسؤولية تمرير هوية الدردشة/الجلسة الحالية إلى حد اكتشاف الإضافة حتى
تكشف أداة `message` المشتركة السطح الصحيح المملوك للقناة للدور الحالي.

أما بالنسبة إلى مساعدات التنفيذ المملوكة للقناة، فيجب على الإضافات المضمّنة إبقاء وقت تشغيل
التنفيذ داخل وحدات الامتداد الخاصة بها. لم تعد النواة تملك
أوقات تشغيل إجراءات الرسائل لـ Discord أو Slack أو Telegram أو WhatsApp تحت `src/agents/tools`.
نحن لا ننشر مسارات فرعية منفصلة من `plugin-sdk/*-action-runtime`، ويجب على
الإضافات المضمّنة استيراد كود وقت التشغيل المحلي الخاص بها مباشرة من
الوحدات المملوكة لامتدادها.

وينطبق الحد نفسه على وصلات SDK المسماة بأسماء المزوّدين عمومًا: يجب ألا
تستورد النواة أشرطة راحة خاصة بالقنوات لـ Slack أو Discord أو Signal أو
WhatsApp أو امتدادات مشابهة. إذا احتاجت النواة إلى سلوك ما، فإما أن تستهلك
شريط `api.ts` / `runtime-api.ts` الخاص بالإضافة المضمّنة نفسها، أو
ترقّي الحاجة إلى قدرة عامة ضيقة في SDK المشترك.

وبالنسبة إلى الاستطلاعات تحديدًا، هناك مساران للتنفيذ:

- `outbound.sendPoll` هو خط الأساس المشترك للقنوات التي تناسب النموذج
  العام للاستطلاع
- `actions.handleAction("poll")` هو المسار المفضل لدلالات الاستطلاع
  الخاصة بالقناة أو معلمات الاستطلاع الإضافية

تؤجل النواة الآن التحليل المشترك للاستطلاع إلى ما بعد رفض توجيه استطلاع الإضافة
للإجراء، حتى تتمكن معالجات الاستطلاع المملوكة للإضافة من قبول حقول استطلاع
خاصة بالقناة دون أن يمنعها محلل الاستطلاع العام أولًا.

راجع [خط أنابيب التحميل](#load-pipeline) للحصول على تسلسل بدء التشغيل الكامل.

## نموذج ملكية القدرات

يعامل OpenClaw الإضافة الأصلية على أنها حد الملكية لـ **شركة** أو
**ميزة**، وليس كحقيبة مختلطة من عمليات التكامل غير المرتبطة.

وهذا يعني:

- يجب أن تملك إضافة الشركة عادةً جميع الأسطح المواجهة لـ OpenClaw الخاصة بتلك الشركة
- يجب أن تملك إضافة الميزة عادةً سطح الميزة الكامل الذي تقدمه
- يجب أن تستهلك القنوات القدرات المشتركة في النواة بدلًا من إعادة تنفيذ
  سلوك المزوّد بصورة مخصصة

أمثلة:

- تملك الإضافة المضمّنة `openai` سلوك مزوّد نماذج OpenAI وسلوك
  OpenAI للكلام + الصوت الفوري + فهم الوسائط + توليد الصور
- تملك الإضافة المضمّنة `elevenlabs` سلوك كلام ElevenLabs
- تملك الإضافة المضمّنة `microsoft` سلوك كلام Microsoft
- تملك الإضافة المضمّنة `google` سلوك مزوّد نماذج Google بالإضافة إلى
  سلوك Google لفهم الوسائط + توليد الصور + البحث على الويب
- تملك الإضافة المضمّنة `firecrawl` سلوك جلب الويب لـ Firecrawl
- تملك الإضافات المضمّنة `minimax` و`mistral` و`moonshot` و`zai`
  واجهاتها الخلفية لفهم الوسائط
- تملك الإضافة `voice-call` كميزة: نقل المكالمات، والأدوات،
  وCLI، والمسارات، وجسر تدفق الوسائط لـ Twilio، لكنها تستهلك قدرات الكلام المشتركة
  بالإضافة إلى النسخ الفوري والصوت الفوري بدلًا من استيراد إضافات المورّدين مباشرة

الحالة النهائية المقصودة هي:

- يعيش OpenAI في إضافة واحدة حتى إذا امتد عبر نماذج النص، والكلام، والصور،
  والفيديو مستقبلًا
- يمكن لمورّد آخر أن يفعل الشيء نفسه لمساحة سطحه الخاصة
- لا تهتم القنوات بأي إضافة مورّد تملك المزوّد؛ فهي تستهلك العقد
  المشترك للقدرات الذي تكشفه النواة

وهذا هو التمييز الأساسي:

- **plugin** = حد الملكية
- **capability** = عقد النواة الذي يمكن لعدة إضافات تنفيذه أو استهلاكه

لذلك إذا أضاف OpenClaw مجالًا جديدًا مثل الفيديو، فالسؤال الأول ليس
"أي مزوّد يجب أن يرمّز التعامل مع الفيديو مباشرة؟" بل السؤال الأول هو
"ما هو عقد قدرة الفيديو في النواة؟" وبمجرد وجود هذا العقد، تستطيع إضافات
المورّدين التسجيل مقابله ويمكن لإضافات القنوات/الميزات استهلاكه.

إذا لم تكن القدرة موجودة بعد، فالخطوة الصحيحة عادة هي:

1. تعريف القدرة المفقودة في النواة
2. كشفها عبر واجهة برمجة الإضافة/وقت التشغيل بطريقة مكتوبة الأنواع
3. ربط القنوات/الميزات بهذه القدرة
4. السماح لإضافات المورّدين بتسجيل التنفيذات

يبقي هذا الملكية صريحة مع تجنب سلوك في النواة يعتمد على
مورّد واحد أو مسار كود خاص بإضافة واحدة.

### طبقات القدرات

استخدم هذا النموذج الذهني عند تقرير مكان الكود:

- **طبقة قدرات النواة**: التنسيق المشترك، والسياسات، والاحتياط، وقواعد
  دمج الإعدادات، ودلالات التسليم، والعقود المكتوبة الأنواع
- **طبقة إضافة المورّد**: واجهات برمجة التطبيقات الخاصة بالمورّد، والمصادقة،
  وكتالوجات النماذج، وتوليف الكلام، وتوليد الصور، وواجهات الفيديو
  المستقبلية، ونقاط نهاية الاستخدام
- **طبقة إضافة القناة/الميزة**: تكامل Slack/Discord/voice-call/... إلخ
  الذي يستهلك قدرات النواة ويعرضها على سطح ما

على سبيل المثال، يتبع تحويل النص إلى كلام هذا الشكل:

- تملك النواة سياسة TTS وقت الرد، وترتيب الاحتياط، والتفضيلات، وتسليم القناة
- تملك `openai` و`elevenlabs` و`microsoft` تنفيذات التوليف
- تستهلك `voice-call` مساعد وقت تشغيل TTS للهاتف

ويجب تفضيل النمط نفسه للقدرات المستقبلية.

### مثال على إضافة شركة متعددة القدرات

يجب أن تبدو إضافة الشركة مترابطة من الخارج. إذا كان لدى OpenClaw عقود مشتركة
للنماذج، والكلام، والنسخ الفوري، والصوت الفوري، وفهم الوسائط،
وتوليد الصور، وتوليد الفيديو، وجلب الويب، والبحث على الويب،
فيمكن للمورّد امتلاك جميع أسطحه في مكان واحد:

```ts
import type { OpenClawPluginDefinition } from "openclaw/plugin-sdk/plugin-entry";
import {
  describeImageWithModel,
  transcribeOpenAiCompatibleAudio,
} from "openclaw/plugin-sdk/media-understanding";

const plugin: OpenClawPluginDefinition = {
  id: "exampleai",
  name: "ExampleAI",
  register(api) {
    api.registerProvider({
      id: "exampleai",
      // auth/model catalog/runtime hooks
    });

    api.registerSpeechProvider({
      id: "exampleai",
      // vendor speech config — implement the SpeechProviderPlugin interface directly
    });

    api.registerMediaUnderstandingProvider({
      id: "exampleai",
      capabilities: ["image", "audio", "video"],
      async describeImage(req) {
        return describeImageWithModel({
          provider: "exampleai",
          model: req.model,
          input: req.input,
        });
      },
      async transcribeAudio(req) {
        return transcribeOpenAiCompatibleAudio({
          provider: "exampleai",
          model: req.model,
          input: req.input,
        });
      },
    });

    api.registerWebSearchProvider(
      createPluginBackedWebSearchProvider({
        id: "exampleai-search",
        // credential + fetch logic
      }),
    );
  },
};

export default plugin;
```

المهم ليس أسماء المساعدات الدقيقة. المهم هو الشكل:

- إضافة واحدة تملك سطح المورّد
- لا تزال النواة تملك عقود القدرات
- تستهلك إضافات القنوات والميزات مساعدات `api.runtime.*`، لا كود المورّد
- يمكن لاختبارات العقد تأكيد أن الإضافة سجلت القدرات التي
  تدّعي أنها تملكها

### مثال قدرة: فهم الفيديو

يعامل OpenClaw بالفعل فهم الصورة/الصوت/الفيديو على أنه قدرة مشتركة
واحدة. وينطبق نموذج الملكية نفسه هنا:

1. تعرّف النواة عقد فهم الوسائط
2. تسجل إضافات المورّدين `describeImage` و`transcribeAudio` و
   `describeVideo` حسب الاقتضاء
3. تستهلك إضافات القنوات والميزات السلوك المشترك في النواة بدلًا من
   الربط المباشر مع كود المورّد

هذا يمنع إدخال افتراضات الفيديو الخاصة بمزوّد واحد في النواة. فالإضافة تملك
سطح المورّد؛ والنواة تملك عقد القدرة وسلوك الاحتياط.

ويتبع توليد الفيديو بالفعل هذا التسلسل نفسه: تملك النواة عقد
القدرة المكتوب الأنواع ومساعد وقت التشغيل، وتسجل إضافات المورّدين
تنفيذات `api.registerVideoGenerationProvider(...)` مقابله.

هل تحتاج إلى قائمة تحقق عملية للنشر؟ راجع
[دليل القدرات](/ar/plugins/architecture).

## العقود وفرضها

سطح واجهة برمجة الإضافة مكتوب الأنواع ومركزي عمدًا في
`OpenClawPluginApi`. يعرّف هذا العقد نقاط التسجيل المدعومة و
مساعدات وقت التشغيل التي يمكن أن تعتمد عليها الإضافة.

سبب أهمية هذا:

- يحصل مؤلفو الإضافات على معيار داخلي مستقر واحد
- يمكن للنواة رفض الملكية المكررة، مثل تسجيل إضافتين لمعرّف
  المزوّد نفسه
- يمكن لبدء التشغيل إظهار تشخيصات عملية للتسجيل غير الصحيح
- يمكن لاختبارات العقد فرض ملكية الإضافات المضمّنة ومنع الانحراف الصامت

هناك طبقتان من الفرض:

1. **فرض التسجيل وقت التشغيل**
   يتحقق سجل الإضافات من التسجيلات أثناء تحميل الإضافات. أمثلة:
   تؤدي معرّفات المزوّدين المكررة، ومعرّفات مزوّدي الكلام المكررة، والتسجيلات
   غير الصحيحة إلى تشخيصات للإضافة بدلًا من سلوك غير معرّف.
2. **اختبارات العقد**
   تُلتقط الإضافات المضمّنة في سجلات العقود أثناء تشغيل الاختبارات حتى
   يتمكن OpenClaw من تأكيد الملكية صراحة. ويُستخدم هذا اليوم من أجل
   مزوّدي النماذج، ومزوّدي الكلام، ومزوّدي البحث على الويب، وملكية التسجيل
   المضمّن.

الأثر العملي هو أن OpenClaw يعرف مسبقًا أي إضافة تملك أي
سطح. وهذا يسمح للنواة والقنوات بالتركيب السلس لأن الملكية
معلنة ومكتوبة الأنواع وقابلة للاختبار بدلًا من أن تكون ضمنية.

### ما الذي ينتمي إلى عقد

العقود الجيدة للإضافات تكون:

- مكتوبة الأنواع
- صغيرة
- خاصة بالقدرة
- مملوكة للنواة
- قابلة لإعادة الاستخدام من عدة إضافات
- قابلة للاستهلاك من القنوات/الميزات دون معرفة بالمورّد

العقود السيئة للإضافات تكون:

- سياسة خاصة بالمورّد مخفية في النواة
- مخارج خاصة لإضافة واحدة تتجاوز السجل
- كود القناة الذي يصل مباشرة إلى تنفيذ خاص بمورّد
- كائنات وقت تشغيل مخصصة ليست جزءًا من `OpenClawPluginApi` أو
  `api.runtime`

عند الشك، ارفع مستوى التجريد: عرّف القدرة أولًا، ثم دع الإضافات
تتصل بها.

## نموذج التنفيذ

تعمل إضافات OpenClaw الأصلية **داخل العملية** مع Gateway. وهي ليست
ضمن صندوق حماية. وللإضافة الأصلية المحمّلة حد الثقة نفسه على مستوى العملية مثل
كود النواة.

الآثار:

- يمكن للإضافة الأصلية تسجيل أدوات ومعالجات شبكة وخطافات وخدمات
- يمكن لخلل في الإضافة الأصلية أن يعطّل البوابة أو يزعزع استقرارها
- تعادل الإضافة الأصلية الخبيثة تنفيذ كود عشوائي داخل
  عملية OpenClaw

تكون الحزم المتوافقة أكثر أمانًا افتراضيًا لأن OpenClaw يعاملها حاليًا على
أنها حزم بيانات/محتوى. وفي الإصدارات الحالية، يعني هذا غالبًا
المهارات المضمّنة.

استخدم قوائم السماح ومسارات التثبيت/التحميل الصريحة للإضافات غير المضمّنة. وتعامل مع
إضافات مساحة العمل على أنها كود وقت تطوير، لا إعدادات إنتاج افتراضية.

بالنسبة إلى أسماء حزم الإضافات المضمّنة في مساحة العمل، أبقِ معرّف الإضافة
مرتبطًا باسم npm: `@openclaw/<id>` افتراضيًا، أو لاحقة نوعية معتمدة مثل
`-provider` أو `-plugin` أو `-speech` أو `-sandbox` أو `-media-understanding` عندما
تكشف الحزمة عمدًا عن دور إضافة أضيق.

ملاحظة ثقة مهمة:

- `plugins.allow` يثق في **معرّفات الإضافات**، وليس في مصدر الأصل.
- تقوم إضافة مساحة عمل لها المعرف نفسه كإضافة مضمّنة بإخفاء
  النسخة المضمّنة عمدًا عندما تكون تلك الإضافة في مساحة العمل مفعّلة/مدرجة في قائمة السماح.
- هذا طبيعي ومفيد للتطوير المحلي، واختبار التصحيحات، والإصلاحات العاجلة.

## حد التصدير

يُصدّر OpenClaw القدرات، لا وسائل الراحة التنفيذية.

أبقِ تسجيل القدرات عامًا. وقلّص تصديرات المساعدات غير التعاقدية:

- المسارات الفرعية الخاصة بمساعدات الإضافات المضمّنة
- المسارات الفرعية الخاصة بسباكة وقت التشغيل غير المقصود أن تكون واجهة برمجة عامة
- المساعدات المريحة الخاصة بالمورّد
- مساعدات الإعداد/التهيئة التي تُعد تفاصيل تنفيذ

لا تزال بعض المسارات الفرعية لمساعدات الإضافات المضمّنة موجودة في خريطة تصدير SDK
المولّدة من أجل التوافق وصيانة الإضافات المضمّنة. وتشمل الأمثلة الحالية
`plugin-sdk/feishu` و`plugin-sdk/feishu-setup` و`plugin-sdk/zalo` و
`plugin-sdk/zalo-setup` وعدة وصلات `plugin-sdk/matrix*`. تعامل مع هذه
على أنها تصديرات محجوزة لتفاصيل التنفيذ، وليست النمط الموصى به في SDK
للإضافات الخارجية الجديدة.

## خط أنابيب التحميل

عند بدء التشغيل، يفعل OpenClaw تقريبًا ما يلي:

1. اكتشاف جذور الإضافات المرشحة
2. قراءة بيانات الإضافات الأصلية أو بيانات الحزم المتوافقة وبيانات الحزم
3. رفض المرشحين غير الآمنين
4. تطبيع إعدادات الإضافات (`plugins.enabled` و`allow` و`deny` و`entries` و
   `slots` و`load.paths`)
5. تقرير التمكين لكل مرشح
6. تحميل الوحدات الأصلية المفعلة عبر jiti
7. استدعاء خطافات `register(api)` الأصلية (أو `activate(api)` — اسم بديل قديم) وجمع التسجيلات في سجل الإضافات
8. كشف السجل لأسطح الأوامر/وقت التشغيل

<Note>
`activate` اسم بديل قديم لـ `register` — يحل أداة التحميل أيّهما موجود (`def.register ?? def.activate`) ويستدعيه في النقطة نفسها. تستخدم جميع الإضافات المضمّنة `register`؛ وفضّل `register` للإضافات الجديدة.
</Note>

تحدث بوابات الأمان **قبل** تنفيذ وقت التشغيل. يُحظر المرشحون
عندما يخرج المدخل من جذر الإضافة، أو يكون المسار قابلاً للكتابة عالميًا، أو
تبدو ملكية المسار مريبة للإضافات غير المضمّنة.

### سلوك البيان أولًا

البيان هو مصدر الحقيقة لمستوى التحكم. يستخدمه OpenClaw من أجل:

- تحديد الإضافة
- اكتشاف القنوات/المهارات/مخطط الإعدادات المعلنة أو قدرات الحزمة
- التحقق من `plugins.entries.<id>.config`
- تعزيز تسميات/عناصر نائبة في Control UI
- إظهار بيانات التثبيت/الكتالوج

بالنسبة إلى الإضافات الأصلية، تكون وحدة وقت التشغيل هي جزء مستوى البيانات. وهي تسجل
السلوك الفعلي مثل الخطافات، أو الأدوات، أو الأوامر، أو تدفقات المزوّد.

### ما الذي تخزنه أداة التحميل مؤقتًا

يحتفظ OpenClaw بذاكرات مؤقتة قصيرة داخل العملية من أجل:

- نتائج الاكتشاف
- بيانات سجل البيان
- سجلات الإضافات المحمّلة

تقلل هذه الذاكرات المؤقتة من بدء التشغيل المتقطع وتكرار عبء الأوامر. ومن الآمن
اعتبارها ذاكرات مؤقتة قصيرة العمر للأداء، لا تخزينًا دائمًا.

ملاحظة أداء:

- عيّن `OPENCLAW_DISABLE_PLUGIN_DISCOVERY_CACHE=1` أو
  `OPENCLAW_DISABLE_PLUGIN_MANIFEST_CACHE=1` لتعطيل هذه الذاكرات المؤقتة.
- اضبط نوافذ الذاكرة المؤقتة باستخدام `OPENCLAW_PLUGIN_DISCOVERY_CACHE_MS` و
  `OPENCLAW_PLUGIN_MANIFEST_CACHE_MS`.

## نموذج السجل

لا تعدّل الإضافات المحمّلة متغيرات عشوائية في النواة مباشرة. بل تسجّل في
سجل إضافات مركزي.

يتتبع السجل ما يلي:

- سجلات الإضافات (الهوية، والمصدر، والأصل، والحالة، والتشخيصات)
- الأدوات
- الخطافات القديمة والخطافات المكتوبة الأنواع
- القنوات
- المزوّدون
- معالجات Gateway RPC
- مسارات HTTP
- مسجّلات CLI
- الخدمات الخلفية
- الأوامر المملوكة للإضافة

ثم تقرأ ميزات النواة من هذا السجل بدلًا من التحدث إلى وحدات الإضافات
مباشرة. وهذا يبقي التحميل في اتجاه واحد:

- وحدة الإضافة -> التسجيل في السجل
- وقت تشغيل النواة -> استهلاك السجل

وهذا الفصل مهم لقابلية الصيانة. فهو يعني أن معظم أسطح النواة لا
تحتاج إلا إلى نقطة تكامل واحدة: "اقرأ السجل"، لا "خصّص كل وحدة إضافة على حدة".

## نداءات ربط المحادثة

يمكن للإضافات التي تربط محادثة أن تتفاعل عند حل الموافقة.

استخدم `api.onConversationBindingResolved(...)` لتلقي نداء بعد اعتماد أو رفض
طلب الربط:

```ts
export default {
  id: "my-plugin",
  register(api) {
    api.onConversationBindingResolved(async (event) => {
      if (event.status === "approved") {
        // A binding now exists for this plugin + conversation.
        console.log(event.binding?.conversationId);
        return;
      }

      // The request was denied; clear any local pending state.
      console.log(event.request.conversation.conversationId);
    });
  },
};
```

حقول حمولة النداء:

- `status`: `"approved"` أو `"denied"`
- `decision`: `"allow-once"` أو `"allow-always"` أو `"deny"`
- `binding`: الربط الذي تم حله للطلبات المعتمدة
- `request`: ملخص الطلب الأصلي، وتلميح الفصل، ومعرّف المرسِل،
  وبيانات المحادثة

هذا النداء للإشعار فقط. فهو لا يغيّر من يُسمح له بربط
محادثة، ويعمل بعد انتهاء معالجة الموافقة في النواة.

## خطافات وقت تشغيل المزوّد

أصبح لدى إضافات المزوّد الآن طبقتان:

- بيانات البيان: `providerAuthEnvVars` للبحث الرخيص عن مصادقة المزوّد
  عبر متغيرات البيئة قبل تحميل وقت التشغيل، و`channelEnvVars` للبحث الرخيص
  عن إعداد/بيئة القناة قبل تحميل وقت التشغيل، بالإضافة إلى `providerAuthChoices`
  للتسميات الرخيصة لخيارات الإعداد/المصادقة وبيانات أعلام CLI قبل تحميل وقت التشغيل
- خطافات وقت الإعداد: `catalog` / الاسم القديم `discovery` بالإضافة إلى `applyConfigDefaults`
- خطافات وقت التشغيل: `normalizeModelId`, `normalizeTransport`,
  `normalizeConfig`,
  `applyNativeStreamingUsageCompat`, `resolveConfigApiKey`,
  `resolveSyntheticAuth`, `resolveExternalAuthProfiles`,
  `shouldDeferSyntheticProfileAuth`,
  `resolveDynamicModel`, `prepareDynamicModel`, `normalizeResolvedModel`,
  `contributeResolvedModelCompat`, `capabilities`,
  `normalizeToolSchemas`, `inspectToolSchemas`,
  `resolveReasoningOutputMode`, `prepareExtraParams`, `createStreamFn`,
  `wrapStreamFn`, `resolveTransportTurnState`,
  `resolveWebSocketSessionPolicy`, `formatApiKey`, `refreshOAuth`,
  `buildAuthDoctorHint`, `matchesContextOverflowError`,
  `classifyFailoverReason`, `isCacheTtlEligible`,
  `buildMissingAuthMessage`, `suppressBuiltInModel`, `augmentModelCatalog`,
  `isBinaryThinking`, `supportsXHighThinking`,
  `resolveDefaultThinkingLevel`, `isModernModelRef`, `prepareRuntimeAuth`,
  `resolveUsageAuth`, `fetchUsageSnapshot`, `createEmbeddingProvider`,
  `buildReplayPolicy`,
  `sanitizeReplayHistory`, `validateReplayTurns`, `onModelSelected`

لا تزال OpenClaw تملك الحلقة العامة للوكيل، والفشل الاحتياطي، ومعالجة النصوص،
وسياسة الأدوات. وتمثل هذه الخطافات سطح الامتداد للسلوك الخاص بالمزوّد دون
الحاجة إلى نقل استدلال مخصص بالكامل.

استخدم البيان `providerAuthEnvVars` عندما يكون لدى المزوّد بيانات اعتماد
تعتمد على متغيرات البيئة ويجب أن تراها مسارات المصادقة/الحالة/منتقي النماذج العامة
دون تحميل وقت تشغيل الإضافة. واستخدم البيان `providerAuthChoices` عندما
تحتاج أسطح CLI الخاصة بالإعداد/اختيار المصادقة إلى معرفة معرّف اختيار المزوّد،
وتسميات المجموعات، وربط المصادقة البسيط بعلم واحد دون تحميل وقت تشغيل المزوّد.
وأبقِ `envVars` في وقت تشغيل المزوّد لتلميحات موجهة للمشغّل مثل تسميات الإعداد أو
متغيرات إعداد OAuth client-id/client-secret.

استخدم البيان `channelEnvVars` عندما تكون لدى القناة مصادقة أو إعداد يعتمد
على البيئة ويجب أن تراه بدائل shell-env العامة، أو فحوصات config/status، أو
مطالبات الإعداد، دون تحميل وقت تشغيل القناة.

### ترتيب الخطافات واستخدامها

بالنسبة إلى إضافات النموذج/المزوّد، يستدعي OpenClaw الخطافات تقريبًا بهذا الترتيب.
ويمثل عمود "متى يُستخدم" دليل القرار السريع.

| #   | الخطاف                            | ما الذي يفعله                                                                                                   | متى يُستخدم                                                                                                                                 |
| --- | --------------------------------- | --------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | `catalog`                         | نشر إعدادات المزوّد داخل `models.providers` أثناء توليد `models.json`                                          | عندما يملك المزوّد كتالوجًا أو إعدادات افتراضية لـ base URL                                                                                |

يفحص `normalizeModelId` و`normalizeTransport` و`normalizeConfig` أولًا
إضافة المزوّد المطابقة، ثم ينتقل إلى إضافات مزوّدين أخرى قادرة على الخطافات
حتى يغيّر أحدها فعليًا معرّف النموذج أو النقل/الإعدادات. وهذا يبقي
التحويلات البديلة/التوافقية للمزوّدين تعمل دون الحاجة إلى معرفة
المستدعي أي إضافة مضمّنة تملك إعادة الكتابة. وإذا لم تعِد أي خطافات مزوّد
كتابة إدخال إعدادات مدعوم من Google-family، فلا يزال مطبّع إعدادات Google
المضمّن يطبّق تنظيف التوافق ذاك.

إذا احتاج المزوّد إلى بروتوكول wire مخصص بالكامل أو منفذ طلبات مخصص بالكامل،
فهذه فئة مختلفة من الامتدادات. فهذه الخطافات مخصصة لسلوك المزوّد
الذي لا يزال يعمل على حلقة الاستدلال العادية في OpenClaw.

### مثال المزوّد

```ts
api.registerProvider({
  id: "example-proxy",
  label: "Example Proxy",
  auth: [],
  catalog: {
    order: "simple",
    run: async (ctx) => {
      const apiKey = ctx.resolveProviderApiKey("example-proxy").apiKey;
      if (!apiKey) {
        return null;
      }
      return {
        provider: {
          baseUrl: "https://proxy.example.com/v1",
          apiKey,
          api: "openai-completions",
          models: [{ id: "auto", name: "Auto" }],
        },
      };
    },
  },
  resolveDynamicModel: (ctx) => ({
    id: ctx.modelId,
    name: ctx.modelId,
    provider: "example-proxy",
    api: "openai-completions",
    baseUrl: "https://proxy.example.com/v1",
    reasoning: false,
    input: ["text"],
    cost: { input: 0, output: 0, cacheRead: 0, cacheWrite: 0 },
    contextWindow: 128000,
    maxTokens: 8192,
  }),
  prepareRuntimeAuth: async (ctx) => {
    const exchanged = await exchangeToken(ctx.apiKey);
    return {
      apiKey: exchanged.token,
      baseUrl: exchanged.baseUrl,
      expiresAt: exchanged.expiresAt,
    };
  },
  resolveUsageAuth: async (ctx) => {
    const auth = await ctx.resolveOAuthToken();
    return auth ? { token: auth.token } : null;
  },
  fetchUsageSnapshot: async (ctx) => {
    return await fetchExampleProxyUsage(ctx.token, ctx.timeoutMs, ctx.fetchFn);
  },
});
```

### أمثلة مضمّنة

- يستخدم Anthropic كلًا من `resolveDynamicModel` و`capabilities` و`buildAuthDoctorHint` و
  `resolveUsageAuth` و`fetchUsageSnapshot` و`isCacheTtlEligible` و
  `resolveDefaultThinkingLevel` و`applyConfigDefaults` و`isModernModelRef` و
  `wrapStreamFn` لأنه يملك توافق Claude 4.6 الأمامي،
  وتلميحات عائلة المزوّد، وإرشادات إصلاح المصادقة، وتكامل نقطة نهاية الاستخدام،
  وأهلية ذاكرة الموجّه المؤقتة، والإعدادات الافتراضية الواعية بالمصادقة،
  وسياسة التفكير الافتراضية/التكيفية لـ Claude، وتشكيل التدفق الخاص بـ Anthropic من أجل
  رؤوس beta، و`/fast` / `serviceTier`، و`context1m`.
- تبقى مساعدات تدفق Anthropic الخاصة بـ Claude داخل وصلة
  `api.ts` / `contract-api.ts` العامة الخاصة بالإضافة المضمّنة في الوقت الحالي. يصدّر هذا
  سطح الحزمة `wrapAnthropicProviderStream` و`resolveAnthropicBetas` و
  `resolveAnthropicFastMode` و`resolveAnthropicServiceTier` والبناة الأدنى
  مستوى لأغلفة Anthropic بدلًا من توسيع SDK العام حول قواعد
  رؤوس beta الخاصة بمزوّد واحد.
- يستخدم OpenAI `resolveDynamicModel` و`normalizeResolvedModel` و
  `capabilities` بالإضافة إلى `buildMissingAuthMessage` و`suppressBuiltInModel` و
  `augmentModelCatalog` و`supportsXHighThinking` و`isModernModelRef`
  لأنه يملك التوافق الأمامي لـ GPT-5.4، والتطبيع المباشر من
  `openai-completions` إلى `openai-responses`، وتلميحات المصادقة الواعية بـ Codex،
  وإخفاء Spark، وصفوف قائمة OpenAI الاصطناعية، وسياسة التفكير /
  النموذج الحي لـ GPT-5؛ وتملك عائلة التدفق `openai-responses-defaults`
  أغلفة OpenAI Responses الأصلية المشتركة لرؤوس الإسناد،
  و`/fast`/`serviceTier`، وإسهاب النص، وبحث Codex الأصلي على الويب،
  وتشكيل حمولة reasoning-compat، وإدارة سياق Responses.
- يستخدم OpenRouter `catalog` بالإضافة إلى `resolveDynamicModel` و
  `prepareDynamicModel` لأن المزوّد تمريري وقد يكشف معرّفات
  نماذج جديدة قبل تحديث كتالوج OpenClaw الثابت؛ كما يستخدم
  `capabilities` و`wrapStreamFn` و`isCacheTtlEligible` لإبقاء
  رؤوس الطلب الخاصة بالمزوّد، وبيانات التوجيه، وترقيعات الاستدلال،
  وسياسة ذاكرة الموجّه المؤقتة خارج النواة. وتأتي سياسة replay الخاصة به من
  عائلة `passthrough-gemini`، بينما تملك عائلة التدفق `openrouter-thinking`
  حقن الاستدلال عبر الوكيل وتخطي النماذج غير المدعومة و`auto`.
- يستخدم GitHub Copilot `catalog` و`auth` و`resolveDynamicModel` و
  `capabilities` بالإضافة إلى `prepareRuntimeAuth` و`fetchUsageSnapshot` لأنه
  يحتاج إلى تسجيل دخول الجهاز المملوك للمزوّد، وسلوك احتياط النموذج، وخصائص
  نصوص Claude، وتبادل رمز GitHub إلى رمز Copilot، ونقطة نهاية
  استخدام مملوكة للمزوّد.
- يستخدم OpenAI Codex `catalog` و`resolveDynamicModel` و
  `normalizeResolvedModel` و`refreshOAuth` و`augmentModelCatalog` بالإضافة إلى
  `prepareExtraParams` و`resolveUsageAuth` و`fetchUsageSnapshot` لأنه
  لا يزال يعمل على عمليات نقل OpenAI الأساسية لكنه يملك تطبيع
  النقل/base URL، وسياسة احتياط تحديث OAuth، وخيار النقل الافتراضي،
  وصفوف كتالوج Codex الاصطناعية، وتكامل نقطة نهاية استخدام ChatGPT؛ وهو
  يشارك عائلة التدفق `openai-responses-defaults` نفسها مع OpenAI المباشر.
- يستخدم Google AI Studio وGemini CLI OAuth كلًا من `resolveDynamicModel` و
  `buildReplayPolicy` و`sanitizeReplayHistory` و
  `resolveReasoningOutputMode` و`wrapStreamFn` و`isModernModelRef` لأن
  عائلة replay `google-gemini` تملك احتياط التوافق الأمامي لـ Gemini 3.1،
  والتحقق الأصلي من replay في Gemini، وتنقية replay التمهيدية، ووضع
  reasoning-output الموسوم، ومطابقة النموذج الحديث، بينما
  تملك عائلة التدفق `google-thinking` تطبيع حمولة تفكير Gemini؛ كما يستخدم
  Gemini CLI OAuth أيضًا `formatApiKey` و`resolveUsageAuth` و
  `fetchUsageSnapshot` لتنسيق الرمز، وتحليل الرمز، وربط نقطة نهاية الحصة.
- يستخدم Anthropic Vertex `buildReplayPolicy` عبر
  عائلة replay `anthropic-by-model` حتى يبقى تنظيف replay الخاص بـ Claude
  مقصورًا على معرّفات Claude بدلًا من كل نقل `anthropic-messages`.
- يستخدم Amazon Bedrock `buildReplayPolicy` و`matchesContextOverflowError` و
  `classifyFailoverReason` و`resolveDefaultThinkingLevel` لأنه يملك
  تصنيف أخطاء Bedrock الخاصة بالخنق/عدم الجاهزية/تجاوز السياق
  لحركة Anthropic-on-Bedrock؛ بينما لا تزال سياسة replay الخاصة به تشترك في
  حاجز `anthropic-by-model` الخاص بـ Claude فقط.
- يستخدم OpenRouter وKilocode وOpencode وOpencode Go `buildReplayPolicy`

## مساعدات وقت التشغيل

يمكن للإضافات الوصول إلى مساعدات مختارة من النواة عبر `api.runtime`. بالنسبة إلى TTS:

```ts
const clip = await api.runtime.tts.textToSpeech({
  text: "Hello from OpenClaw",
  cfg: api.config,
});

const result = await api.runtime.tts.textToSpeechTelephony({
  text: "Hello from OpenClaw",
  cfg: api.config,
});

const voices = await api.runtime.tts.listVoices({
  provider: "elevenlabs",
  cfg: api.config,
});
```

ملاحظات:

- يعيد `textToSpeech` حمولة إخراج TTS العادية في النواة لأسطح الملفات/الملاحظات الصوتية.
- يستخدم إعدادات `messages.tts` في النواة واختيار المزوّد.
- يعيد مخزن PCM للصوت + معدل العينة. يجب على الإضافات إعادة أخذ العينات/الترميز للمزوّدات.
- `listVoices` اختيارية لكل مزوّد. استخدمها لمنتقيات الأصوات أو تدفقات الإعداد المملوكة للمورّد.
- يمكن أن تتضمن قوائم الأصوات بيانات وصفية أغنى مثل اللغة والمنطقة، والنوع، ووسوم الشخصية لمنتقيات واعية بالمزوّد.
- يدعم OpenAI وElevenLabs الهاتفية اليوم. ولا يدعمها Microsoft.

يمكن للإضافات أيضًا تسجيل مزوّدي الكلام عبر `api.registerSpeechProvider(...)`.

```ts
api.registerSpeechProvider({
  id: "acme-speech",
  label: "Acme Speech",
  isConfigured: ({ config }) => Boolean(config.messages?.tts),
  synthesize: async (req) => {
    return {
      audioBuffer: Buffer.from([]),
      outputFormat: "mp3",
      fileExtension: ".mp3",
      voiceCompatible: false,
    };
  },
});
```

ملاحظات:

- أبقِ سياسة TTS، والاحتياط، وتسليم الرد في النواة.
- استخدم مزوّدي الكلام لسلوك التوليف المملوك للمورّد.
- تُطبَّع قيمة الإدخال القديمة `edge` الخاصة بـ Microsoft إلى معرّف المزوّد `microsoft`.
- نموذج الملكية المفضل هو الموجّه للشركة: يمكن لإضافة مورّد واحدة أن تملك
  النص والكلام والصورة ومزوّدي الوسائط المستقبليين مع إضافة OpenClaw لعقود
  تلك القدرات.

بالنسبة إلى فهم الصور/الصوت/الفيديو، تسجل الإضافات مزوّد
فهم وسائط واحدًا مكتوب الأنواع بدلًا من كيس مفتاح/قيمة عام:

```ts
api.registerMediaUnderstandingProvider({
  id: "google",
  capabilities: ["image", "audio", "video"],
  describeImage: async (req) => ({ text: "..." }),
  transcribeAudio: async (req) => ({ text: "..." }),
  describeVideo: async (req) => ({ text: "..." }),
});
```

ملاحظات:

- أبقِ التنسيق، والاحتياط، والإعدادات، وربط القنوات في النواة.
- أبقِ سلوك المورّد في إضافة المزوّد.
- يجب أن يبقى التوسع الإضافي مكتوب الأنواع: أساليب اختيارية جديدة، وحقول
  نتائج اختيارية جديدة، وقدرات اختيارية جديدة.
- يتبع توليد الفيديو بالفعل النمط نفسه:
  - تملك النواة عقد القدرة ومساعد وقت التشغيل
  - تسجل إضافات المورّدين `api.registerVideoGenerationProvider(...)`
  - تستهلك إضافات الميزة/القناة `api.runtime.videoGeneration.*`

بالنسبة إلى مساعدات وقت التشغيل لفهم الوسائط، يمكن للإضافات استدعاء:

```ts
const image = await api.runtime.mediaUnderstanding.describeImageFile({
  filePath: "/tmp/inbound-photo.jpg",
  cfg: api.config,
  agentDir: "/tmp/agent",
});

const video = await api.runtime.mediaUnderstanding.describeVideoFile({
  filePath: "/tmp/inbound-video.mp4",
  cfg: api.config,
});
```

وبالنسبة إلى نسخ الصوت، يمكن للإضافات استخدام وقت تشغيل فهم الوسائط أو
الاسم البديل الأقدم STT:

```ts
const { text } = await api.runtime.mediaUnderstanding.transcribeAudioFile({
  filePath: "/tmp/inbound-audio.ogg",
  cfg: api.config,
  // Optional when MIME cannot be inferred reliably:
  mime: "audio/ogg",
});
```

ملاحظات:

- يمثل `api.runtime.mediaUnderstanding.*` السطح المشترك المفضل لـ
  فهم الصور/الصوت/الفيديو.
- يستخدم إعدادات الصوت الخاصة بفهم الوسائط في النواة (`tools.media.audio`) وترتيب احتياط المزوّد.
- يعيد `{ text: undefined }` عندما لا يتم إنتاج إخراج نسخ (على سبيل المثال إدخال متخطى/غير مدعوم).
- يبقى `api.runtime.stt.transcribeAudioFile(...)` اسمًا بديلًا للتوافق.

يمكن للإضافات أيضًا إطلاق عمليات subagent في الخلفية عبر `api.runtime.subagent`:

```ts
const result = await api.runtime.subagent.run({
  sessionKey: "agent:main:subagent:search-helper",
  message: "Expand this query into focused follow-up searches.",
  provider: "openai",
  model: "gpt-4.1-mini",
  deliver: false,
});
```

ملاحظات:

- إن `provider` و`model` تجاوزات اختيارية لكل تشغيل، وليسا تغييرات دائمة للجلسة.
- لا يحترم OpenClaw حقول التجاوز هذه إلا للجهات الموثوقة.
- بالنسبة إلى عمليات الاحتياط المملوكة للإضافة، يجب على المشغلين تفعيل
  `plugins.entries.<id>.subagent.allowModelOverride: true`.
- استخدم `plugins.entries.<id>.subagent.allowedModels` لتقييد الإضافات الموثوقة
  بأهداف `provider/model` قانونية محددة، أو `"*"` للسماح صراحة بأي هدف.
- لا تزال تشغيلات subagent الخاصة بالإضافات غير الموثوقة تعمل، لكن تُرفض طلبات التجاوز بدلًا من الاحتياط الصامت.

بالنسبة إلى البحث على الويب، يمكن للإضافات استهلاك مساعد وقت التشغيل المشترك بدلًا من
الوصول إلى ربط أداة الوكيل:

```ts
const providers = api.runtime.webSearch.listProviders({
  config: api.config,
});

const result = await api.runtime.webSearch.search({
  config: api.config,
  args: {
    query: "OpenClaw plugin runtime helpers",
    count: 5,
  },
});
```

يمكن للإضافات أيضًا تسجيل مزوّدي البحث على الويب عبر
`api.registerWebSearchProvider(...)`.

ملاحظات:

- أبقِ اختيار المزوّد، وحل بيانات الاعتماد، ودلالات الطلب المشتركة في النواة.
- استخدم مزوّدي البحث على الويب لعمليات النقل الخاصة بالمورّد.
- يمثل `api.runtime.webSearch.*` السطح المشترك المفضل لإضافات الميزات/القنوات التي تحتاج إلى سلوك بحث من دون الاعتماد على غلاف أداة الوكيل.

## مسارات Gateway HTTP

يمكن للإضافات كشف نقاط نهاية HTTP باستخدام `api.registerHttpRoute(...)`.

```ts
api.registerHttpRoute({
  path: "/acme/webhook",
  auth: "plugin",
  match: "exact",
  handler: async (_req, res) => {
    res.statusCode = 200;
    res.end("ok");
    return true;
  },
});
```

حقول المسار:

- `path`: مسار التوجيه تحت خادم HTTP الخاص بـ Gateway.
- `auth`: مطلوب. استخدم `"gateway"` لطلب مصادقة Gateway العادية، أو `"plugin"` لمصادقة/تحقق webhook المُدار بواسطة الإضافة.
- `match`: اختياري. `"exact"` (الافتراضي) أو `"prefix"`.
- `replaceExisting`: اختياري. يسمح للإضافة نفسها باستبدال تسجيل مسارها الحالي.
- `handler`: أعد `true` عندما يتولى المسار معالجة الطلب.

ملاحظات:

- تمت إزالة `api.registerHttpHandler(...)` وسيؤدي إلى خطأ تحميل إضافة. استخدم `api.registerHttpRoute(...)` بدلًا منه.
- يجب أن تعلن مسارات الإضافة `auth` صراحة.
- تُرفض تعارضات `path + match` الدقيقة ما لم يكن `replaceExisting: true`، ولا يمكن لإضافة أن تستبدل مسار إضافة أخرى.
- تُرفض المسارات المتداخلة ذات مستويات `auth` المختلفة. أبقِ سلاسل السقوط `exact`/`prefix` على مستوى المصادقة نفسه فقط.
- لا تتلقى مسارات `auth: "plugin"` **نطاقات وقت تشغيل المشغّل** تلقائيًا. فهي مخصصة لـ webhooks/التحقق من التوقيع المُدار بواسطة الإضافة، وليس لاستدعاءات مساعدة Gateway ذات الامتيازات.
- تعمل مسارات `auth: "gateway"` داخل نطاق وقت تشغيل طلب Gateway، لكن هذا النطاق محافظ عمدًا:
  - تبقي مصادقة bearer بالمفتاح المشترك (`gateway.auth.mode = "token"` / `"password"`) نطاقات وقت تشغيل مسارات الإضافة مثبّتة عند `operator.write`، حتى إذا أرسل المستدعي `x-openclaw-scopes`
  - تحترم أوضاع HTTP الموثوقة الحاملة للهوية (مثل `trusted-proxy` أو `gateway.auth.mode = "none"` على مدخل خاص) قيمة `x-openclaw-scopes` فقط عندما يكون الرأس موجودًا صراحة
  - إذا غاب `x-openclaw-scopes` في طلبات مسارات الإضافة الحاملة للهوية تلك، يعود نطاق وقت التشغيل إلى `operator.write`
- القاعدة العملية: لا تفترض أن مسار إضافة موثّق عبر gateway هو سطح إدارة ضمني. إذا كان مسارك يحتاج إلى سلوك خاص بالمشرف فقط، فاطلب وضع مصادقة حاملًا للهوية ووثّق عقد الرأس `x-openclaw-scopes` الصريح.

- تبقى الأجزاء الداخلية للامتدادات المضمّنة خاصة. يجب على الإضافات الخارجية استخدام
  المسارات الفرعية `openclaw/plugin-sdk/*` فقط. يمكن لكود/اختبارات نواة OpenClaw
  استخدام نقاط الدخول العامة في المستودع تحت جذر حزمة الإضافة مثل `index.js` و`api.js` و
  `runtime-api.js` و`setup-entry.js` والملفات الضيقة النطاق مثل
  `login-qr-api.js`. ولا تستورد أبدًا `src/*` لحزمة إضافة من النواة أو من
  امتداد آخر.
- تقسيم نقطة دخول المستودع:
  يمثّل `<plugin-package-root>/api.js` شريط المساعدات/الأنواع،
  ويمثّل `<plugin-package-root>/runtime-api.js` شريط وقت التشغيل فقط،
  ويمثّل `<plugin-package-root>/index.js` مدخل الإضافة المضمّنة،
  ويمثّل `<plugin-package-root>/setup-entry.js` مدخل إضافة الإعداد.
- أمثلة المزوّدين المضمّنين الحالية:
  - يستخدم Anthropic `api.js` / `contract-api.js` لمساعدات تدفق Claude مثل
    `wrapAnthropicProviderStream`، ومساعدات رؤوس beta، وتحليل `service_tier`.
  - يستخدم OpenAI `api.js` لبناة المزوّد، ومساعدات النموذج الافتراضي، وبناة
    المزوّد الفوري.
  - يستخدم OpenRouter `api.js` لباني المزوّد الخاص به بالإضافة إلى مساعدات
    الإعداد/التهيئة، بينما يظل `register.runtime.js` قادرًا على إعادة تصدير
    مساعدات `plugin-sdk/provider-stream` العامة للاستخدام المحلي داخل المستودع.
- تفضّل نقاط الدخول العامة المحمّلة عبر الواجهة
  لقطة إعدادات وقت التشغيل النشطة عندما تكون موجودة، ثم تعود إلى
  ملف الإعدادات المحلول على القرص عندما لا يكون OpenClaw قد بدأ بعد
  في تقديم لقطة وقت تشغيل.
- تظل البدائيات العامة المشتركة هي عقد SDK العام المفضل. ولا تزال توجد
  مجموعة توافق صغيرة محجوزة من وصلات المساعدة ذات العلامة القنوية للإضافات المضمّنة.
  تعامل معها على أنها وصلات صيانة/توافق للإضافات المضمّنة، لا أهداف استيراد جديدة
  لأطراف ثالثة؛ ويجب أن تهبط العقود الجديدة المشتركة بين القنوات على
  المسارات العامة `plugin-sdk/*` أو على الأشرطة المحلية للإضافة `api.js` /
  `runtime-api.js`.

ملاحظة التوافق:

- تجنب شريط `openclaw/plugin-sdk` الجذري في الكود الجديد.
- فضّل البدائيات المستقرة الضيقة أولًا. فالمسارات الفرعية الأحدث للإعداد/الاقتران/الرد/
  والتغذية الراجعة/العقد/الوارد/الترابط/الأوامر/secret-input/webhook/infra/
  وقائمة السماح/الحالة/أداة الرسائل هي العقد المقصود للعمل الجديد على
  الإضافات المضمّنة والخارجية.
  يجب أن ينتمي التحليل/المطابقة للأهداف إلى `openclaw/plugin-sdk/channel-targets`.
  وتنتمي بوابات إجراءات الرسائل ومساعدات message-id الخاصة بالتفاعلات إلى
  `openclaw/plugin-sdk/channel-actions`.
- لا تُعد أشرطة المساعدة الخاصة بالامتدادات المضمّنة مستقرة افتراضيًا. إذا كانت
  المساعدة مطلوبة فقط لامتداد مضمّن، فأبقها خلف
  الوصلة المحلية لذلك الامتداد `api.js` أو `runtime-api.js` بدلًا من ترقيتها إلى
  `openclaw/plugin-sdk/<extension>`.
- يجب أن تكون وصلات المساعدة المشتركة الجديدة عامة، لا تحمل علامة قناة. ينتمي
  تحليل الأهداف المشترك إلى `openclaw/plugin-sdk/channel-targets`؛ أما
  الأجزاء الداخلية الخاصة بالقناة فتبقى خلف الوصلة المحلية `api.js` أو `runtime-api.js`
  للإضافة المالكة.
- توجد المسارات الفرعية الخاصة بالقدرات مثل `image-generation` و
  `media-understanding` و`speech` لأن الإضافات المضمّنة/الأصلية تستخدمها
  اليوم. ولا يعني وجودها بحد ذاته أن كل مساعدة مصدّرة هي عقد خارجي
  مجمّد على المدى الطويل.

## مخططات أداة الرسائل

يجب أن تملك الإضافات مساهمات مخطط `describeMessageTool(...)` الخاصة بالقنوات.
أبقِ الحقول الخاصة بالمزوّد في الإضافة، لا في النواة المشتركة.

وبالنسبة إلى أجزاء المخطط المحمولة المشتركة، أعد استخدام المساعدات العامة المصدّرة عبر
`openclaw/plugin-sdk/channel-actions`:

- `createMessageToolButtonsSchema()` لحِمولات نمط شبكة الأزرار
- `createMessageToolCardSchema()` لحِمولات البطاقات المنظمة

إذا كان شكل المخطط لا معنى له إلا لمزوّد واحد، فعرّفه في
مصدر تلك الإضافة بدلًا من ترقيته إلى SDK المشترك.

## حل أهداف القنوات

يجب أن تملك إضافات القنوات دلالات الأهداف الخاصة بالقناة. أبقِ مضيف
الإرسال الخارجي عامًا واستخدم سطح مهايئ المراسلة لقواعد المزوّد:

- يقرر `messaging.inferTargetChatType({ to })` ما إذا كان الهدف المطبّع
  يجب التعامل معه على أنه `direct` أو `group` أو `channel` قبل البحث في الدليل.
- يخبر `messaging.targetResolver.looksLikeId(raw, normalized)` النواة
  ما إذا كان الإدخال يجب أن يتجاوز مباشرة إلى حل شبيه بالمعرّف بدلًا من البحث في الدليل.
- يمثّل `messaging.targetResolver.resolveTarget(...)` احتياط الإضافة عندما
  تحتاج النواة إلى حل نهائي مملوك للمزوّد بعد التطبيع أو بعد
  فشل الدليل.
- تملك `messaging.resolveOutboundSessionRoute(...)` بناء مسار الجلسة
  الخارجي الخاص بالمزوّد بعد حل الهدف.

التقسيم الموصى به:

- استخدم `inferTargetChatType` لقرارات الفئة التي يجب أن تحدث قبل
  البحث في الأقران/المجموعات.
- استخدم `looksLikeId` لفحوصات "عامل هذا كمعرّف هدف صريح/أصلي".
- استخدم `resolveTarget` كاحتياط تطبيع خاص بالمزوّد، لا من أجل
  بحث واسع في الدليل.
- أبقِ المعرّفات الأصلية للمزوّد مثل معرّفات الدردشة، ومعرّفات السلاسل، وJIDs،
  والمعرّفات، ومعرّفات الغرف داخل قيم `target` أو المعلمات الخاصة بالمزوّد، لا في
  حقول SDK عامة.

## الأدلة المعتمدة على الإعدادات

يجب أن تبقي الإضافات التي تستمد إدخالات الدليل من الإعدادات هذا المنطق داخل
الإضافة وأن تعيد استخدام المساعدات المشتركة من
`openclaw/plugin-sdk/directory-runtime`.

استخدم هذا عندما تحتاج قناة إلى أقران/مجموعات معتمدة على الإعدادات مثل:

- أقران الرسائل المباشرة المدفوعة بقائمة السماح
- خرائط القنوات/المجموعات المهيأة
- بدائل دليل ثابتة بنطاق الحساب

لا تتعامل المساعدات المشتركة في `directory-runtime` إلا مع العمليات العامة:

- تصفية الاستعلام
- تطبيق الحدود
- إزالة التكرار/مساعدات التطبيع
- بناء `ChannelDirectoryEntry[]`

يجب أن يبقى فحص الحساب الخاص بالقناة وتطبيع المعرّفات داخل
تنفيذ الإضافة.

## كتالوجات المزوّدين

يمكن لإضافات المزوّد تحديد كتالوجات نماذج للاستدلال باستخدام
`registerProvider({ catalog: { run(...) { ... } } })`.

يعيد `catalog.run(...)` الشكل نفسه الذي يكتبه OpenClaw إلى
`models.providers`:

- `{ provider }` لإدخال مزوّد واحد
- `{ providers }` لعدة إدخالات مزوّدين

استخدم `catalog` عندما تملك الإضافة معرّفات نماذج خاصة بالمزوّد، أو إعدادات
base URL افتراضية، أو بيانات نماذج تعتمد على المصادقة.

يتحكم `catalog.order` في وقت دمج كتالوج الإضافة بالنسبة إلى
المزوّدين الضمنيين المضمّنين في OpenClaw:

- `simple`: مزوّدون عاديون يعتمدون على API key أو البيئة
- `profile`: مزوّدون يظهرون عند وجود ملفات تعريف مصادقة
- `paired`: مزوّدون يُنشئون عدة إدخالات مزوّدين مرتبطة
- `late`: المرور الأخير، بعد بقية المزوّدين الضمنيين

يفوز المزوّد اللاحق عند تعارض المفتاح، لذلك يمكن للإضافات عمدًا تجاوز
إدخال مزوّد مضمّن يملك معرّف المزوّد نفسه.

التوافق:

- لا يزال `discovery` يعمل كاسم بديل قديم
- إذا سُجِّل كل من `catalog` و`discovery`، يستخدم OpenClaw `catalog`

## فحص القناة للقراءة فقط

إذا كانت إضافتك تسجل قناة، ففضّل تنفيذ
`plugin.config.inspectAccount(cfg, accountId)` إلى جانب `resolveAccount(...)`.

السبب:

- يمثل `resolveAccount(...)` مسار وقت التشغيل. ويُسمح له بافتراض أن بيانات الاعتماد
  قد جرى تفعيلها بالكامل، ويمكنه الفشل بسرعة عند غياب الأسرار المطلوبة.
- يجب ألا تحتاج مسارات الأوامر للقراءة فقط مثل `openclaw status` و
  `openclaw status --all` و`openclaw channels status` و`openclaw channels resolve` وعمليات
  doctor/config repair إلى تفعيل بيانات اعتماد وقت التشغيل فقط من أجل
  وصف الإعدادات.

السلوك الموصى به لـ `inspectAccount(...)`:

- أعد حالة وصفية للحساب فقط.
- احتفظ بـ `enabled` و`configured`.
- أدرج حقول مصدر/حالة بيانات الاعتماد عند الحاجة، مثل:
  - `tokenSource`, `tokenStatus`
  - `botTokenSource`, `botTokenStatus`
  - `appTokenSource`, `appTokenStatus`
  - `signingSecretSource`, `signingSecretStatus`
- لا تحتاج إلى إعادة قيم الرموز الخام فقط للإبلاغ عن
  التوفر للقراءة فقط. يكفي إرجاع `tokenStatus: "available"` (وحقل المصدر
  المطابق).
- استخدم `configured_unavailable` عندما تكون بيانات الاعتماد مهيأة عبر SecretRef لكنها
  غير متاحة في مسار الأمر الحالي.

يسمح ذلك للأوامر المخصصة للقراءة فقط بالإبلاغ عن "مهيأ لكن غير متاح في مسار هذا الأمر"
بدلًا من التعطل أو الإبلاغ خطأ أن الحساب غير مهيأ.

## حزم الحزم

قد يتضمن دليل الإضافة ملف `package.json` مع `openclaw.extensions`:

```json
{
  "name": "my-pack",
  "openclaw": {
    "extensions": ["./src/safety.ts", "./src/tools.ts"],
    "setupEntry": "./src/setup-entry.ts"
  }
}
```

يصبح كل إدخال إضافة. وإذا أدرجت الحزمة عدة امتدادات، يصبح معرّف الإضافة
`name/<fileBase>`.

إذا كانت إضافتك تستورد تبعيات npm، فثبّتها في ذلك الدليل حتى
يصبح `node_modules` متاحًا (`npm install` / `pnpm install`).

حاجز أمان: يجب أن يبقى كل إدخال في `openclaw.extensions` داخل دليل الإضافة
بعد حل الروابط الرمزية. وتُرفض الإدخالات التي تهرب من دليل الحزمة.

ملاحظة أمان: يثبت `openclaw plugins install` تبعيات الإضافة باستخدام
`npm install --omit=dev --ignore-scripts` (من دون سكربتات دورة الحياة، ومن دون تبعيات تطوير وقت التشغيل). أبقِ أشجار تبعيات الإضافة
"JS/TS خالصة" وتجنب الحزم التي تتطلب بناء `postinstall`.

اختياري: يمكن أن يشير `openclaw.setupEntry` إلى وحدة خفيفة خاصة بالإعداد فقط.
عندما يحتاج OpenClaw إلى أسطح إعداد لإضافة قناة معطلة، أو
عندما تكون إضافة القناة مفعلة لكنها لا تزال غير مهيأة، فإنه يحمّل `setupEntry`
بدلًا من مدخل الإضافة الكامل. وهذا يبقي بدء التشغيل والإعداد أخف
عندما يكون مدخل الإضافة الرئيسي لديك يربط أيضًا أدوات أو خطافات أو كودًا
آخر خاصًا بوقت التشغيل فقط.

اختياري: يمكن لـ `openclaw.startup.deferConfiguredChannelFullLoadUntilAfterListen`
إدخال إضافة قناة في مسار `setupEntry` نفسه خلال مرحلة
بدء التشغيل السابقة للاستماع في البوابة، حتى عندما تكون القناة مهيأة بالفعل.

استخدم هذا فقط عندما يغطي `setupEntry` بالكامل سطح بدء التشغيل الذي يجب أن يوجد
قبل أن تبدأ البوابة في الاستماع. وعمليًا، يعني هذا أن مدخل الإعداد
يجب أن يسجل كل قدرة مملوكة للقناة يعتمد عليها بدء التشغيل، مثل:

- تسجيل القناة نفسه
- أي مسارات HTTP يجب أن تكون متاحة قبل أن تبدأ البوابة في الاستماع
- أي أساليب Gateway أو أدوات أو خدمات يجب أن توجد أثناء النافذة نفسها

إذا كان مدخلك الكامل لا يزال يملك أي قدرة مطلوبة عند بدء التشغيل، فلا تفعّل
هذا العلم. أبقِ الإضافة على السلوك الافتراضي ودع OpenClaw يحمّل
المدخل الكامل أثناء بدء التشغيل.

يمكن للقنوات المضمّنة أيضًا نشر مساعدات سطح عقد خاصة بالإعداد فقط يمكن للنواة
استشارتها قبل تحميل وقت تشغيل القناة الكامل. وسطح الترقية الحالي للإعداد هو:

- `singleAccountKeysToMove`
- `namedAccountPromotionKeys`
- `resolveSingleAccountPromotionTarget(...)`

تستخدم النواة هذا السطح عندما تحتاج إلى ترقية إعدادات قناة قديمة بحساب واحد
إلى `channels.<id>.accounts.*` دون تحميل مدخل الإضافة الكامل.
وتُعد Matrix المثال المضمّن الحالي: فهي تنقل فقط مفاتيح المصادقة/التمهيد إلى
حساب مُرقّى مسمّى عندما تكون الحسابات المسماة موجودة بالفعل، ويمكنها
الحفاظ على مفتاح حساب افتراضي غير قانوني مهيأ بدلًا من إنشاء
`accounts.default` دائمًا.

تبقي مهايئات ترقيع الإعداد هذه اكتشاف سطح عقد الإضافة المضمّنة كسولًا. فيبقى
وقت الاستيراد خفيفًا؛ ويُحمَّل سطح الترقية عند أول استخدام فقط بدلًا من
إعادة دخول بدء تشغيل القناة المضمّنة عند استيراد الوحدة.

عندما تتضمن أسطح بدء التشغيل هذه أساليب Gateway RPC، فأبقها على
بادئة خاصة بالإضافة. وتظل مساحات أسماء الإدارة الأساسية (`config.*`,
`exec.approvals.*`, `wizard.*`, `update.*`) محجوزة وتحل دائمًا إلى
`operator.admin`، حتى إذا طلبت الإضافة نطاقًا أضيق.

مثال:

```json
{
  "name": "@scope/my-channel",
  "openclaw": {
    "extensions": ["./index.ts"],
    "setupEntry": "./setup-entry.ts",
    "startup": {
      "deferConfiguredChannelFullLoadUntilAfterListen": true
    }
  }
}
```

### بيانات كتالوج القنوات الوصفية

يمكن لإضافات القنوات الإعلان عن بيانات الإعداد/الاكتشاف الوصفية عبر `openclaw.channel` و
تلميحات التثبيت عبر `openclaw.install`. وهذا يبقي بيانات الكتالوج خارج النواة.

مثال:

```json
{
  "name": "@openclaw/nextcloud-talk",
  "openclaw": {
    "extensions": ["./index.ts"],
    "channel": {
      "id": "nextcloud-talk",
      "label": "Nextcloud Talk",
      "selectionLabel": "Nextcloud Talk (self-hosted)",
      "docsPath": "/channels/nextcloud-talk",
      "docsLabel": "nextcloud-talk",
      "blurb": "Self-hosted chat via Nextcloud Talk webhook bots.",
      "order": 65,
      "aliases": ["nc-talk", "nc"]
    },
    "install": {
      "npmSpec": "@openclaw/nextcloud-talk",
      "localPath": "<bundled-plugin-local-path>",
      "defaultChoice": "npm"
    }
  }
}
```

حقول `openclaw.channel` المفيدة إلى جانب المثال الأدنى:

- `detailLabel`: تسمية ثانوية لأسطح الكتالوج/الحالة الأكثر غنى
- `docsLabel`: تجاوز نص رابط الوثائق
- `preferOver`: معرّفات الإضافة/القناة ذات الأولوية الأقل التي يجب أن يتفوق عليها إدخال الكتالوج هذا
- `selectionDocsPrefix`, `selectionDocsOmitLabel`, `selectionExtras`: عناصر تحكم النسخ الخاصة بسطح الاختيار
- `markdownCapable`: يضع علامة على القناة على أنها قادرة على Markdown لقرارات التنسيق الخارجي
- `exposure.configured`: إخفاء القناة من أسطح عرض القنوات المهيأة عندما تكون قيمته `false`
- `exposure.setup`: إخفاء القناة من منتقيات الإعداد/التهيئة التفاعلية عندما تكون قيمته `false`
- `exposure.docs`: وضع علامة على القناة على أنها داخلية/خاصة لأسطح التنقل في الوثائق
- `showConfigured` / `showInSetup`: أسماء بديلة قديمة لا تزال مقبولة للتوافق؛ ويفضّل `exposure`
- `quickstartAllowFrom`: إدخال القناة في تدفق `allowFrom` السريع القياسي
- `forceAccountBinding`: فرض ربط حساب صريح حتى عند وجود حساب واحد فقط
- `preferSessionLookupForAnnounceTarget`: تفضيل البحث في الجلسة عند حل أهداف الإعلان

يمكن لـ OpenClaw أيضًا دمج **كتالوجات قنوات خارجية** (على سبيل المثال، تصدير
سجل MPM). ضع ملف JSON في أحد المسارات التالية:

- `~/.openclaw/mpm/plugins.json`
- `~/.openclaw/mpm/catalog.json`
- `~/.openclaw/plugins/catalog.json`

أو وجّه `OPENCLAW_PLUGIN_CATALOG_PATHS` (أو `OPENCLAW_MPM_CATALOG_PATHS`) إلى
ملف JSON واحد أو أكثر (مفصولة بفواصل/فواصل منقوطة/`PATH`). يجب أن يحتوي كل ملف
على `{ "entries": [ { "name": "@scope/pkg", "openclaw": { "channel": {...}, "install": {...} } } ] }`. يقبل المحلل أيضًا `"packages"` أو `"plugins"` كأسماء بديلة قديمة للمفتاح `"entries"`.

## إضافات محرك السياق

تملك إضافات محرك السياق تنسيق سياق الجلسة لعمليات الإدخال،
والتجميع، والضغط. سجّلها من إضافتك باستخدام
`api.registerContextEngine(id, factory)`، ثم حدّد المحرك النشط باستخدام
`plugins.slots.contextEngine`.

استخدم هذا عندما تحتاج إضافتك إلى استبدال خط أنابيب السياق الافتراضي أو
توسيعه بدلًا من مجرد إضافة بحث في الذاكرة أو خطافات.

```ts
import { buildMemorySystemPromptAddition } from "openclaw/plugin-sdk/core";

export default function (api) {
  api.registerContextEngine("lossless-claw", () => ({
    info: { id: "lossless-claw", name: "Lossless Claw", ownsCompaction: true },
    async ingest() {
      return { ingested: true };
    },
    async assemble({ messages, availableTools, citationsMode }) {
      return {
        messages,
        estimatedTokens: 0,
        systemPromptAddition: buildMemorySystemPromptAddition({
          availableTools: availableTools ?? new Set(),
          citationsMode,
        }),
      };
    },
    async compact() {
      return { ok: true, compacted: false };
    },
  }));
}
```

إذا كان محركك **لا** يملك خوارزمية الضغط، فأبقِ `compact()`
منفذًا وفوّضه صراحة:

```ts
import {
  buildMemorySystemPromptAddition,
  delegateCompactionToRuntime,
} from "openclaw/plugin-sdk/core";

export default function (api) {
  api.registerContextEngine("my-memory-engine", () => ({
    info: {
      id: "my-memory-engine",
      name: "My Memory Engine",
      ownsCompaction: false,
    },
    async ingest() {
      return { ingested: true };
    },
    async assemble({ messages, availableTools, citationsMode }) {
      return {
        messages,
        estimatedTokens: 0,
        systemPromptAddition: buildMemorySystemPromptAddition({
          availableTools: availableTools ?? new Set(),
          citationsMode,
        }),
      };
    },
    async compact(params) {
      return await delegateCompactionToRuntime(params);
    },
  }));
}
```

## إضافة قدرة جديدة

عندما تحتاج إضافة إلى سلوك لا يناسب واجهة برمجة التطبيقات الحالية، فلا
تتجاوز نظام الإضافات بتوغّل خاص. أضف القدرة المفقودة.

التسلسل الموصى به:

1. تعريف عقد النواة
   قرر ما السلوك المشترك الذي يجب أن تملكه النواة: السياسة، والاحتياط، ودمج الإعدادات،
   ودورة الحياة، والدلالات المواجهة للقناة، وشكل مساعد وقت التشغيل.
2. إضافة أسطح تسجيل/وقت تشغيل مكتوبة الأنواع للإضافة
   وسّع `OpenClawPluginApi` و/أو `api.runtime` بأصغر
   سطح قدرة مكتوب الأنواع مفيد.
3. ربط مستهلكي النواة + القناة/الميزة
   يجب أن تستهلك القنوات وإضافات الميزات القدرة الجديدة عبر النواة،
   لا باستيراد تنفيذ مورّد مباشرة.
4. تسجيل تنفيذات المورّد
   تسجل إضافات المورّدين بعد ذلك واجهاتها الخلفية مقابل القدرة.
5. إضافة تغطية للعقد
   أضف اختبارات حتى تبقى الملكية وشكل التسجيل صريحين مع الزمن.

هكذا يبقى OpenClaw صاحب رأي دون أن يصبح مشفّرًا مباشرة وفق
رؤية مزوّد واحد. راجع [دليل القدرات](/ar/plugins/architecture)
للاطلاع على قائمة ملفات عملية ومثال تطبيقي.

### قائمة التحقق الخاصة بالقدرات

عند إضافة قدرة جديدة، يجب أن يلمس التنفيذ عادةً هذه
الأسطح معًا:

- أنواع عقد النواة في `src/<capability>/types.ts`
- العداء/مساعد وقت التشغيل في النواة في `src/<capability>/runtime.ts`
- سطح تسجيل واجهة برمجة الإضافة في `src/plugins/types.ts`
- ربط سجل الإضافات في `src/plugins/registry.ts`
- كشف وقت تشغيل الإضافة في `src/plugins/runtime/*` عندما تحتاج
  إضافات الميزة/القناة إلى استهلاكها
- مساعدات الالتقاط/الاختبار في `src/test-utils/plugin-registration.ts`
- تأكيدات الملكية/العقد في `src/plugins/contracts/registry.ts`
- وثائق المشغّل/الإضافة في `docs/`

إذا كان أحد هذه الأسطح مفقودًا، فغالبًا ما يكون ذلك علامة على أن القدرة
لم تُدمج بالكامل بعد.

### قالب القدرة

النمط الأدنى:

```ts
// core contract
export type VideoGenerationProviderPlugin = {
  id: string;
  label: string;
  generateVideo: (req: VideoGenerationRequest) => Promise<VideoGenerationResult>;
};

// plugin API
api.registerVideoGenerationProvider({
  id: "openai",
  label: "OpenAI",
  async generateVideo(req) {
    return await generateOpenAiVideo(req);
  },
});

// shared runtime helper for feature/channel plugins
const clip = await api.runtime.videoGeneration.generate({
  prompt: "Show the robot walking through the lab.",
  cfg,
});
```

نمط اختبار العقد:

```ts
expect(findVideoGenerationProviderIdsForPlugin("openai")).toEqual(["openai"]);
```

ويبقي ذلك القاعدة بسيطة:

- تملك النواة عقد القدرة + التنسيق
- تملك إضافات المورّدين تنفيذات المورّد
- تستهلك إضافات الميزة/القناة مساعدات وقت التشغيل
- تُبقي اختبارات العقد الملكية صريحة
