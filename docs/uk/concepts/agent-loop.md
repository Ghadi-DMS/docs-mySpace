# Цикл агента (OpenClaw)

Агентний цикл — це повний «реальний» запуск агента: приймання → збирання контексту → інференс моделі →
виконання інструментів → потокові відповіді → збереження. Це авторитетний шлях, який перетворює повідомлення
на дії та фінальну відповідь, зберігаючи узгоджений стан сесії.

У OpenClaw цикл — це один серіалізований запуск на сесію, який видає події життєвого циклу та потокові події,
поки модель міркує, викликає інструменти та передає вихідні дані потоком. У цьому документі пояснюється, як
цей автентичний цикл з’єднано наскрізно.

## Точки входу

- Gateway RPC: `agent` і `agent.wait`.
- CLI: команда `agent`.

## Як це працює (високий рівень)

1. RPC `agent` перевіряє параметри, визначає сесію (sessionKey/sessionId), зберігає метадані сесії, одразу повертає `{ runId, acceptedAt }`.
2. `agentCommand` запускає агента:
   - визначає модель і типові значення thinking/verbose
   - завантажує знімок Skills
   - викликає `runEmbeddedPiAgent` (середовище виконання pi-agent-core)
   - видає **lifecycle end/error**, якщо вбудований цикл цього не зробив
3. `runEmbeddedPiAgent`:
   - серіалізує запуски через черги на рівні сесії та глобальні черги
   - визначає профіль моделі та auth і створює сесію pi
   - підписується на події pi і передає дельти assistant/tool потоком
   - застосовує timeout -> перериває запуск у разі перевищення
   - повертає корисні дані та метадані usage
4. `subscribeEmbeddedPiSession` з’єднує події pi-agent-core з потоком OpenClaw `agent`:
   - події інструментів => `stream: "tool"`
   - дельти помічника => `stream: "assistant"`
   - події життєвого циклу => `stream: "lifecycle"` (`phase: "start" | "end" | "error"`)
5. `agent.wait` використовує `waitForAgentRun`:
   - очікує на **lifecycle end/error** для `runId`
   - повертає `{ status: ok|error|timeout, startedAt, endedAt, error? }`

## Черги та паралелізм

- Запуски серіалізуються для кожного ключа сесії (доріжка сесії) і, за потреби, через глобальну доріжку.
- Це запобігає гонкам інструментів/сесій і зберігає узгоджену історію сесії.
- Канали обміну повідомленнями можуть вибирати режими черги (collect/steer/followup), які підключаються до цієї системи доріжок.
  Див. [Черга команд](/uk/concepts/queue).

## Підготовка сесії та робочого простору

- Робочий простір визначається і створюється; ізольовані запуски можуть перенаправлятися до кореня ізольованого робочого простору.
- Skills завантажуються (або повторно використовуються зі знімка) і впроваджуються в env та prompt.
- Файли bootstrap/context визначаються та впроваджуються у звіт системного prompt.
- Отримується блокування запису сесії; `SessionManager` відкривається і готується до початку потокової передачі.

## Збирання prompt і системний prompt

- Системний prompt будується з базового prompt OpenClaw, Skills prompt, bootstrap-контексту та перевизначень для конкретного запуску.
- Застосовуються обмеження для конкретної моделі та резерв токенів для compaction.
- Див. [Системний prompt](/uk/concepts/system-prompt), щоб зрозуміти, що бачить модель.

## Точки hooks (де можна перехопити виконання)

OpenClaw має дві системи hooks:

- **Внутрішні hooks** (Gateway hooks): скрипти, керовані подіями, для команд і подій життєвого циклу.
- **Plugin hooks**: точки розширення всередині життєвого циклу агента/інструментів і конвеєра gateway.

### Внутрішні hooks (Gateway hooks)

- **`agent:bootstrap`**: виконується під час побудови bootstrap-файлів до того, як буде фіналізовано системний prompt.
  Використовуйте це, щоб додавати або видаляти файли bootstrap-контексту.
- **Command hooks**: `/new`, `/reset`, `/stop` та інші події команд (див. документ Hooks).

Див. [Hooks](/uk/automation/hooks) для налаштування та прикладів.

### Plugin hooks (життєвий цикл агента та gateway)

Вони виконуються всередині циклу агента або конвеєра gateway:

- **`before_model_resolve`**: виконується до сесії (без `messages`), щоб детерміновано перевизначити provider/model до визначення моделі.
- **`before_prompt_build`**: виконується після завантаження сесії (з `messages`), щоб впровадити `prependContext`, `systemPrompt`, `prependSystemContext` або `appendSystemContext` перед надсиланням prompt. Використовуйте `prependContext` для динамічного тексту на кожен хід, а поля системного контексту — для стабільних інструкцій, які мають знаходитися в просторі системного prompt.
- **`before_agent_start`**: застарілий hook сумісності, який може виконуватися в будь-якій фазі; віддавайте перевагу явним hooks вище.
- **`before_agent_reply`**: виконується після вбудованих дій і перед викликом LLM, дозволяючи plugin перебрати хід і повернути синтетичну відповідь або повністю приглушити хід.
- **`agent_end`**: перевіряє фінальний список повідомлень і метадані запуску після завершення.
- **`before_compaction` / `after_compaction`**: спостерігають за циклами compaction або додають до них примітки.
- **`before_tool_call` / `after_tool_call`**: перехоплюють параметри/результати інструментів.
- **`before_install`**: перевіряє результати вбудованого сканування і за потреби блокує встановлення skill або plugin.
- **`tool_result_persist`**: синхронно перетворює результати інструментів перед записом у транскрипт сесії.
- **`message_received` / `message_sending` / `message_sent`**: hooks вхідних і вихідних повідомлень.
- **`session_start` / `session_end`**: межі життєвого циклу сесії.
- **`gateway_start` / `gateway_stop`**: події життєвого циклу gateway.

Правила прийняття рішень hooks для захисту вихідних повідомлень/інструментів:

- `before_tool_call`: `{ block: true }` є остаточним і зупиняє обробники з нижчим пріоритетом.
- `before_tool_call`: `{ block: false }` нічого не робить і не скасовує попереднє блокування.
- `before_install`: `{ block: true }` є остаточним і зупиняє обробники з нижчим пріоритетом.
- `before_install`: `{ block: false }` нічого не робить і не скасовує попереднє блокування.
- `message_sending`: `{ cancel: true }` є остаточним і зупиняє обробники з нижчим пріоритетом.
- `message_sending`: `{ cancel: false }` нічого не робить і не скасовує попереднє скасування.

Див. [Plugin hooks](/uk/plugins/architecture#provider-runtime-hooks) для API hooks і відомостей про реєстрацію.

## Потокова передача та часткові відповіді

- Дельти помічника передаються потоком із pi-agent-core і видаються як події `assistant`.
- Потокова передача блоків може видавати часткові відповіді на `text_end` або `message_end`.
- Потокова передача reasoning може видаватися як окремий потік або як блокові відповіді.
- Див. [Потокова передача](/uk/concepts/streaming) для поведінки chunking і блокових відповідей.

## Виконання інструментів та інструменти обміну повідомленнями

- Події початку/оновлення/завершення інструментів видаються у потоці `tool`.
- Результати інструментів очищуються з огляду на розмір та зображення перед логуванням/виведенням.
- Надсилання через інструменти обміну повідомленнями відстежуються, щоб придушити дубльовані підтвердження від помічника.

## Формування відповіді та приглушення

- Фінальні корисні дані збираються з:
  - тексту помічника (і за потреби reasoning)
  - вбудованих підсумків інструментів (коли verbose увімкнено та це дозволено)
  - тексту помилки помічника, якщо модель повертає помилку
- Точний беззвучний токен `NO_REPLY` / `no_reply` відфільтровується з вихідних
  корисних даних.
- Дублікати інструментів обміну повідомленнями видаляються з фінального списку корисних даних.
- Якщо не залишилося корисних даних, які можна відобразити, і інструмент завершився помилкою, видається
  резервна відповідь про помилку інструмента
  (якщо тільки інструмент обміну повідомленнями вже не надіслав видиму для користувача відповідь).

## Compaction і повторні спроби

- Автоматичний compaction видає потокові події `compaction` і може спричинити повторну спробу.
- Під час повторної спроби буфери в пам’яті та підсумки інструментів скидаються, щоб уникнути дубльованого виведення.
- Див. [Compaction](/uk/concepts/compaction) для конвеєра compaction.

## Потоки подій (зараз)

- `lifecycle`: видається `subscribeEmbeddedPiSession` (і як резервний варіант `agentCommand`)
- `assistant`: потокові дельти з pi-agent-core
- `tool`: потокові події інструментів з pi-agent-core

## Обробка чат-каналу

- Дельти помічника буферизуються в чат-повідомлення `delta`.
- Чат `final` видається на **lifecycle end/error**.

## Тайм-аути

- Типове значення `agent.wait`: 30 с (лише очікування). Перевизначається параметром `timeoutMs`.
- Середовище виконання агента: типове значення `agents.defaults.timeoutSeconds` — 172800 с (48 годин); застосовується таймером переривання в `runEmbeddedPiAgent`.
- Тайм-аут бездіяльності LLM: `agents.defaults.llm.idleTimeoutSeconds` перериває запит до моделі, якщо до завершення вікна бездіяльності не надходять жодні фрагменти відповіді. Встановіть його явно для повільних локальних моделей або provider з reasoning/tool-call; встановіть `0`, щоб вимкнути. Якщо його не встановлено, OpenClaw використовує `agents.defaults.timeoutSeconds`, якщо налаштовано, інакше — `60` с. Запуски, активовані cron, без явного тайм-ауту LLM або агента, вимикають watchdog бездіяльності та покладаються на зовнішній тайм-аут cron.

## Де все може завершитися раніше

- Тайм-аут агента (переривання)
- AbortSignal (скасування)
- Роз’єднання Gateway або тайм-аут RPC
- Тайм-аут `agent.wait` (лише очікування, не зупиняє агента)
